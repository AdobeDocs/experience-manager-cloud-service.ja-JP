# Forms Experience Builder - ベストプラクティス

>[!NOTE]
>
> Forms Experience Builder は、早期導入プログラムで利用できます。 勤務先のアドレスから `aem-forms-ea@adobe.com` にメールを送信して、アクセスをリクエストします。

>[!IMPORTANT]
>
> **ドキュメントは変更される場合があります**：このベストプラクティスガイドは現在製品に対してテスト中で、更新および改訂される可能性があります。 Forms Experience Builder が早期導入プログラム中に進化し続けると、ベストプラクティス、推奨事項、例が変わる可能性があります。

この包括的なガイドでは、Forms Experience Builder を使用して効果的で使いやすいフォームを作成するための実証済みのベストプラクティスを提供します。 これらのプラクティスは、様々な業界やユースケースでの実装の成功やユーザーのフィードバックから派生しています。

## フォームデザインのベストプラクティス

### シンプルに

**必須フィールドから開始**

- ユーザーの負担を減らすために最も必要な情報のみで始める
- ユーザーのニーズとビジネス要件に基づいて徐々に複雑さを追加
- 実際に必要のない情報や使用しない情報を求めないでください
- フォームの設計時に、ユーザーの時間と認識機能の負荷を考慮する

**明確で説明的なラベルの使用**

- 説明ラベルを使用して、フィールドの目的を直ちに明確にします
- ユーザーが理解できない技術的な用語や内部用語を避ける
- フォーム全体で一貫したラベル規則を使用する
- フィールドの要件が明確でない可能性がある場合にコンテキストを提供

**有用なガイダンスの提供**

- 例や書式要件を含む複雑なフィールドのヘルプテキストを含める
- プレースホルダーテキストを使用して、期待される入力形式を表示する
- 許可された形式やサイズ制限など、ファイルのアップロードに関する明確な手順を指定します
- 進行状況インジケーターを使用して複数の手順から成るプロセスのガイド

**徹底的にテスト**

- デプロイメント前にすべてのユーザーのパスとシナリオを検証
- ターゲットオーディエンスからの実際のユーザーでフォームをテストする
- すべての条件付きロジックが期待どおりに動作することを確認します
- エラー処理によって、アクションにつながる明確なフィードバックが提供されるようにします

### プログレッシブ開示

**コンテキストに基づいて関連フィールドを表示**

- ユーザーの選択に関連する場合にのみ追加フィールドを表示します
- 条件付きロジックを使用した認知負荷とフォーム長の削減
- 関連するフィールドを論理セクションにグループ化
- ユーザーが必要と判断するまで詳細オプションを非表示にする

**実装例：**

     「既婚」に等しい場合@spouseInformation のみパネルを表示@maritalStatus るルールを作成 

**ナビゲーションと進行状況のクリア**

- ユーザーが複数手順のフォームのどこにいるかを理解するのに役立ちます
- フォームセクション間の明確なナビゲーションの提供
- 長いフォームの進捗状況インジケーターを表示する
- ユーザーが進行状況を保存し、後で戻ることを許可

## ユーザーエクスペリエンスのベストプラクティス

### モバイルファーストのデザイン

**レスポンシブレイアウトの最適化**

- モバイルユーザーを主な考慮点として使用したフォームのデザイン
- モバイルデバイスへの 1 列レイアウトの使用
- タッチターゲットのサイズが適切であることを確認します（44 px 以上）。
- 様々なデバイスサイズと向きに基づいてフォームをテストする

**タッチ操作対応インタラクション**

- タッチインターフェイスの大きめのボタンと入力フィールドの実装
- 適切な入力タイプを使用して、モバイルキーボードを正しくトリガーします
- タッチデバイスで動作しない、ホバーに依存するインタラクションを避ける
- ユーザーインタラクションに関する明確で視覚的なフィードバックを提供する

### アクセシビリティコンプライアンス

**WCAG 2.1 ガイドライン**

- 包括的なデザインのための web コンテンツアクセシビリティガイドラインに従う
- 適切なカラーコントラスト比を確保します（通常のテキストでは最小 4.5 :1）。
- すべての画像とアイコンに代替テキストを提供する
- 適切な見出し構造とセマンティック HTMLの実装

**キーボードナビゲーション**

- キーボードナビゲーションを使用してすべてのフォーム要素にアクセスできるようにする
- すべてのインタラクティブ要素に明確なフォーカスインジケーターを提供
- フォームフィールドを使用した論理タブ順序の実装
- 複雑なフォームにスキップナビゲーションリンクを含める

**Screen Reader サポート**

- フォームフィールドに適切な ARIA ラベルと説明を使用する
- スクリーンリーダーに読み上げる明確なエラーメッセージを提供する
- 動的コンテンツの変更が適切にアナウンスされるようにします
- 実際のスクリーンリーダーソフトウェアでフォームをテストする

### パフォーマンスの最適化

**読み込み速度**

- 初期バンドルサイズを最小にすることでフォームの読み込み時間を最適化
- 重要でないフォームセクションに対する遅延読み込みの実装
- Web 配信用の画像とアセットの最適化
- 静的リソースに適切なキャッシュ戦略を有効にする

**実行時のパフォーマンス**

- 適切な検証タイミングを使用して、ユーザーエクスペリエンスとパフォーマンスのバランスを取ります
- リアルタイム検証のためのデバウンスを実装して、サーバーリクエストを減らします。
- 頻繁にアクセスされるデータと検証結果をキャッシュする
- 複雑なフォームの条件付きロジック実行の最適化

**自動保存とデータ保護**

- フォームの進行状況の自動保存を実装してデータの損失を防ぐ
- 可能な限りオフライン機能をフォームに入力する
- 失敗した送信の再試行ロジックを含める
- ユーザーのバックアップとしてのデータ書き出しオプションの提供

## ブランド一貫性のベストプラクティス

### 事前に Brand Assetsを準備する

**ブランドテンプレートの作成**

- 組織の視覚的 ID を使用して標準化されたフォームテンプレートを開発する
- 一貫したカラースキーム、タイポグラフィ、レイアウトパターンを含める
- ブランドガイドラインに一致する、再利用可能なコンポーネントを準備
- チーム間で一貫した実装を行うためのドキュメントスタイル標準

**スタイルガイドラインの定義**

- 一貫したフィールドのスタイル設定、ボタンのデザイン、間隔の標準を確立する
- カラーコード、フォントの仕様、レイアウト規則を含むスタイルガイドの作成
- インタラクションパターンとアニメーションガイドラインの定義
- ブランド固有のエラーメッセージとヘルプテキストの準備

**コンポーネントライブラリを作成**

- ブランド ID に一致する、再利用可能なフォームコンポーネントを作成する
- 一貫したナビゲーションパターンとユーザーインターフェイス要素の開発
- フォーム統合のためのブランドアイコン、ロゴ、ビジュアルアセットの準備
- 一般的なフォームタイプ（連絡先、登録、フィードバック）のテンプレートを確立する

### ブランド実装戦略

**一貫性を保つためのスタイルプロンプト**

ブランド固有の手順をプロンプトに含めます。

     次を使用してプロフェッショナルな連絡先フォームを作成：
    - プライマリボタンおよびヘッダーはコーポレートブルー（#003366） 
     – すべてのテキスト要素に Sans フォントファミリーを開く 
    - アクセシビリティコンプライアンスの最小フォントサイズ 
    - フォームセクション間に一貫した 24 px の間隔 
     – 適切なブランド配置のヘッダー内の会社のロゴ 

**テンプレートライブラリの管理**

- 一般的なユースケースに対応するブランド化されたフォームテンプレートのコレクションを管理する
- 一貫性を確保するために、ブランドテンプレートをバージョン管理します
- テンプレートの使用とカスタマイズに関する明確なドキュメントを提供する
- 現在の標準を維持するためのブランドアセットの定期的なレビューと更新

>[!NOTE]
>
>**カスタムコンポーネント**：カスタムブランド要素を実装する前に、組織固有のコンポーネントの使用とそのForms Experience Builder との互換性について開発チームと調整します。

## コンテンツとコミュニケーションのベストプラクティス

### フォーム作成のアプローチ

**2 つのプライマリ方式**

ニーズに最も適した作成方法を選択します。

1. **ゼロから作成**：特定の要件を持つ新しいフォームに最適
2. **インポートと変換**：既存のフォームやドキュメントの最新化に最適です。

**自然言語プロンプト**

- フォームの説明で具体的に詳しく説明します
- 専門用語ではなく、明確でビジネスに焦点を当てた言語を使用する
- フォームの目的とターゲットオーディエンスに関するコンテキストを提供します
- 初期プロンプトに検証およびビジネス・ルール要件を含める

### 増分開発戦略

**シンプルなものから始めて、複雑なものを作る**

- 基本的なフォーム構造と基本的なフィールドから始める
- 検証ルールとビジネスロジックを増分的に追加する
- 次の機能強化に進む前に、各追加をテストします
- 開発の各ステージでユーザーのフィードバックを収集

**増分アプローチの例：**

     手順 1:「名前、メール、メッセージのフィールドを使用した基本的な連絡先フォームの作成」 
     手順 2:「@name を作成し、適切な検証で必須フィールドを@email 定」 
     手順 3:「ユーザーガイダンスのプレースホルダーテキストとヘルプテキストを追加」 
     手順 4:「問い合わせタイプに基づく条件付きロジックの追加」 

### フィールド参照のベストプラクティス

**一貫性のある命名規則**

- 目的に合った、わかりやすいフィールド名を使用します。
- フォーム間で一貫性のある命名パターンを維持する
- フォーム全体で一貫してキャメルケースまたは snake_case を使用する
- チームの一貫性を保つためのフィールド命名規則の文書化

**有効なフィールド参照**

- 既存のフィールドを変更する場合は、`@fieldName` 構文を使用します
- 複雑なフォームで参照するフィールドを明確にする
- 関連するフィールドの変更を 1 つのリクエストにグループ化
- フィールド名を条件付きロジックで使用する前に検証する

## 統合と送信のベストプラクティス

### マルチチャネル送信戦略

**プライマリおよびセカンダリの操作**

- コアビジネスプロセスのプライマリ送信エンドポイントの設定
- 通知、確認およびデータバックアップ用のセカンダリアクションの設定
- 失敗した送信に対するエラー処理と再試行ロジックの実装
- すべての送信状態（成功、失敗、処理）に関するユーザーフィードバックを提供

**統合計画**

- 基本的な送信設定から開始して、増分的に統合を追加する
- 複数のエンドポイントを組み合わせる前に、各統合を個別にテストします
- Document API の要件と認証方法
- データ変換およびマッピング要件の計画

### エラー処理と回復

**使いやすいエラーメッセージ**

- 問題の解決に役立つ、明確で実用的なエラーメッセージを提供する
- ユーザー向けコンテンツでの技術的なエラーコードやシステムメッセージの回避
- プライマリ送信が失敗した場合の代替アクションの提供
- 追加のサポートが必要なユーザーの連絡先情報を含める

**データ保護とリカバリ**

- エラー後のフォーム回復のためのローカルデータ保存の実装
- ユーザーがフォームデータをバックアップとしてダウンロードするためのオプションを提供する
- 送信エラーの監視とアラートの設定
- システムの停止やメンテナンスのためのリカバリ手順を計画

## Performance と Analytics のベストプラクティス

### 監視と最適化

**主要業績評価指標**

- フォームの完了率と放棄ポイントの追跡
- 読み込み時間とユーザー操作パターンの監視
- 検証の有効性とエラー率の測定
- ユーザーのフィードバックと満足度スコアの分析

**継続的改善**

- 最適化の機会を特定するためのフォーム分析の定期的なレビュー
- 様々なフォームデザインとユーザーフローの A/B テスト
- 改善を繰り返すためのユーザーフィードバックの収集と分析
- 業界標準に照らしたパフォーマンス・ベンチマーク

### データ品質と検証

**スマート検証戦略**

- ユーザーからのフィードバックを即座に得るためにリアルタイム検証を実装
- プログレッシブ検証を使用して、複雑な要件に合わせてユーザーをガイドします
- エラーの修正方法を説明する明確な検証メッセージを提供する
- 検証の厳密性とユーザーエクスペリエンスの考慮事項のバランスを取ります

**データの整合性**

- データの一貫性を確保するためにクロスフィールド検証を実装
- 適切な入力タイプと制約を使用して、無効なデータを防ぐ
- 複雑なフィールドのデータフォーマット例とガイダンスの提供
- 送信されたデータの品質と検証の有効性を定期的に監査

## セキュリティとコンプライアンスのベストプラクティス

### データ保護

**プライバシーバイデザイン**

- ビジネス目的に必要な最小限のデータのみを収集します
- 適切なデータ保持および削除ポリシーの導入
- 明確なプライバシー通知と同意メカニズムの提供
- 関連するデータ保護規制（GDPR、CCPA など）へのコンプライアンスを確保します。

**担保の実施**

- すべてのフォーム送信とデータ送信に HTTPS 暗号化を使用
- 適切な入力検証およびサニタイズの実装
- 適切な制限とウイルススキャンを使用して、セキュアファイルのアップロードを設定する
- 定期的なセキュリティ監査と脆弱性の評価

### コンプライアンスに関する考慮事項

**規制要件**

- 業界固有のコンプライアンス要件の理解と導入
- 監査目的でコンプライアンス対策を文書化
- 規制変更とそのフォームへの影響の定期的なレビュー
- コンプライアンス要件と導入に関するスタッフ・トレーニング

**アクセシビリティコンプライアンス**

- アクセシビリティについては、WCAG 2.1 AA ガイドラインに従ってください
- 支援テクノロジーを使用した通常のアクセシビリティテスト
- 障害のあるユーザーを対象としたユーザーテスト
- アクセシビリティ機能とコンプライアンス対策のドキュメント

**基準及び指針**

## 高度なベストプラクティス

### LLM 拡張スマートフィールド

**AI 知識の活用**

- AI の組み込み知識を使用して、包括的なフィールドオプションを実現
- 地理データ、ビジネス分類、業界標準のスマートフィールドをリクエスト
- インテリジェントフィールド母集団の実装によるユーザーエクスペリエンスの向上

     「標準 NAICS 分類を使用した包括的な業界フィールドを作成」

- 自動化とユーザー制御および透明性のバランスを取る

## 検証とテスト

### 包括的なテスト戦略

**マルチレベルテスト**

- 個々のフォームコンポーネントと検証ルールの単体テスト
- 送信エンドポイントとデータフローの統合テスト
- 代表的なユーザーグループによるユーザー受け入れテスト
- 様々な負荷条件下でのパフォーマンステスト

**クロスプラットフォーム検証**

- 様々なブラウザーやバージョンでのフォームのテスト
- 様々なデバイスや画面サイズでのモバイルエクスペリエンスの検証
- 様々な支援テクノロジーを使用したアクセシビリティテスト
- 様々な接続速度に対するネットワーク状態テスト

### 品質指標

**成功指標**

- 業界ベンチマークを上回るフォーム完了率
- エラー率の低さとユーザー満足度の高さ
- 読み込み時間の短縮とレスポンシブなユーザー操作
- バックエンドシステムおよびプロセスとの統合の成功

**常時監視**

- フォームのパフォーマンス指標の定期的なレビューとユーザーからのフィードバック
- 問題のプロアクティブな特定と解決
- フォームの使用状況と有効性のトレンド分析
- 業界標準とベストプラクティスに照らしたベンチマーク
