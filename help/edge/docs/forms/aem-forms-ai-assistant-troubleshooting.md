# Forms Experience Builder - トラブルシューティングガイド

>[!NOTE]
>
> Forms Experience Builder は、早期導入プログラムで利用できます。 勤務先のアドレスから `aem-forms-ea@adobe.com` にメールを送信して、アクセスをリクエストします。

>[!IMPORTANT]
>
> **ドキュメントは変更される場合があります**：このトラブルシューティングガイドは現在製品に対してテスト中で、更新および改訂される可能性があります。 Forms Experience Builder が早期導入プログラム中に進化し続けると、問題、解決策、デバッグ手法が変わる可能性があります。

この包括的なトラブルシューティングガイドは、Forms Experience Builder を使用する際の一般的な問題の特定、診断、解決に役立ちます。 このガイドは、迅速な修正と詳細なソリューションを備えた問題カテゴリ別に整理されています。

## クイックリファレンス – 一般的な問題

| 問題 | 簡易修正 |
|-------|-----------|
| **インターフェイスが読み込まれない** | ブラウザーを更新し、インターネット接続を確認し、早期アクセス権限を確認します |
| **コマンドが機能しない** | `/help` を試すか、スラッシュコマンドの代わりに自然言語を使用します |
| **@fieldName 認識されません** | スペルをチェックし、最初にフィールドが存在することを確認し、フィールド名の構文を確認します |
| **ファイルのアップロードに失敗** | 10MB 未満のPDF/JPG/PNG を使用する、ファイルフォーマットの互換性を確認する |
| **フォームが正しく表示されない** | 具体的には、「レイアウトを修正」ではなく「モバイルに対応」にします |
| **統合に失敗** | API 資格情報と権限を確認し、エンドポイントが使用可能であることを確認します |
| **フォームが送信されない** | 送信アクションの設定と検証ルールの確認 |
| **検証エラーが表示されない** | フィールド検証設定とエラーメッセージ配置の検証 |
| **モバイルレイアウトの問題** | レスポンシブデザイン設定とフィールドサイズのレビュー |
| **フィールドが表示されない** | 条件付きロジックおよび表示ルールの確認 |
| **読み込みの失敗** | ファイル形式の互換性とサイズ制限の検証 |
| **パフォーマンスの問題** | フィールド数を最適化し、不要な検証を削除 |
| **アクセシビリティの問題** | フィールドラベル、ARIA 属性、タブ順序の確認 |

**サポートが必要ですか？「**」と入力した後に具体的な質問を `/help` 力するか、システム管理者に問い合わせて技術的なサポートを依頼します。

## アクセスと認証の問題

### Forms Experience Builder にアクセスできない

**症状：**

- Forms Experience Builder インターフェイスが表示されない
- 「アクセスが拒否されました」などのエラーメッセージ
- エディターにForms Experience Builder アイコンがない

**解決策：**

1. **アーリーアクセスプログラム登録の検証**
   - 早期導入プログラムに承認されていることを確認します
   - リクエストが公式作業メールから送信されたことを確認します
   - アクセスがまだ保留中の場合は `aem-forms-ea@adobe.com` に問い合わせてください

2. **環境設定の確認**
   - お使いの環境でAEM Formsが有効になっていることを確認します
   - サポートされているブラウザー（Chrome、Firefox、Safari、Edge）を使用していることを確認する
   - ブラウザーのキャッシュと Cookie のクリア
   - 干渉する可能性のあるブラウザー拡張機能を無効にする

3. **ユーザー権限の検証**
   - 適切なユーザーの役割と権限があることを確認します。
   - アクセス権限については、システム管理者にお問い合わせください
   - 正しいアカウントでログインしていることを確認してください

### インターフェイス読み込みの問題

**症状：**

- 空白または部分的に読み込まれたインターフェイス
- 完了しない回転負荷指標
- ブラウザーコンソールのJavaScript エラー

**解決策：**

1. **ブラウザーのトラブルシューティング**
   - ページを更新（Ctrl + F5 または Cmd + Shift + R）
   - 別のブラウザーまたは匿名/プライベートモードを試す
   - ブラウザーのアップデートを確認し、利用可能な場合はインストールする
   - 広告ブロッカーとプライバシー拡張機能を一時的に無効にする

2. **ネットワーク接続**
   - 安定したインターネット接続の確認
   - 企業のファイアウォールが必要なドメインをブロックしているかどうかを確認
   - 可能であれば、別のネットワーク接続でテストします
   - ネットワーク設定の問題については、IT サポートにお問い合わせください

3. **キャッシュとストレージの問題**
   - ブラウザーキャッシュとローカルストレージのクリア
   - ブラウザー設定をデフォルトにリセット
   - デバイスの空きディスク容量を確認します
   - 別のデバイスからアクセスしてみてください

## コマンドとインタラクションの問題

### スラッシュコマンドが機能しない

**症状：**

- `/create-form` またはその他のスラッシュコマンドが認識されない
- オートコンプリート候補が表示されない
- コマンドを実行すると、エラーメッセージが表示される

**解決策：**

1. **コマンド構文の検証**
   - 適切なコマンド形式であることを確認します：`/command-name description`
   - コマンド名の入力ミスをチェックする
   - 自然言語を代わりに使用：「お問い合わせフォームを作成する」
   - `/help` でコマンドの可用性を確認してください

2. **コンテキスト固有のコマンド**
   - 正しいエディターコンテキストであることを確認する（ユニバーサルエディターとアダプティブFormsエディター）
   - 一部のコマンドは、特定の環境でのみ機能します
   - コンテキスト要件のコマンド リファレンスを確認する

3. **代替アプローチ**
   - スラッシュコマンドの代わりに自然言語を使用する
   - 複雑なコマンドを小さく単純なリクエストに分割
   - 単一の複雑なコマンドの代わりに、フォームの構築を段階的に試す

### フィールド参照が機能しない

**症状：**

- `@fieldName` 参照が認識されません
- 不明なフィールドに関するエラーメッセージ
- フィールドの変更が正しく適用されない

**解決策：**

1. **フィールド名の検証**
   - フィールド名の正確なスペルをチェックする（大文字と小文字を区別）
   - フィールドを参照する前に、そのフィールドが存在することを確認する
   - ラベルを表示するのではなく、作成されたとおりのフィールド名を使用します
   - フィールド命名規則（camelCase、snake_case など）の確認

2. **フィールドリファレンスの構文**
   - スペースを含まない適切な `@fieldName` 構文を使用する
   - フィールド参照での特殊文字の使用を避ける
   - 非表示の文字またはフォーマットの問題をチェック
   - フィールド参照を手動で再作成してみてください

3. **フィールド参照のデバッグ**
   - 最初にすべての既存のフィールドをリストする：「現在のすべてのフォームフィールドを表示」
   - ルールで参照する前にフィールドを作成する
   - 複合文字を含まない単純なフィールド名を使用する
   - テストフィールドは一度に 1 つずつ参照されます

## フォームの作成とデザインの問題

### フォームが期待どおりに作成されない

**症状：**

- 生成されたフォームにリクエストフィールドがない
- フィールドのタイプまたはレイアウトが正しくありません
- フォーム構造が説明と一致しません

**解決策：**

1. **迅速な特異度の向上**
   - 詳しくは、フォームの説明を参照してください
   - 正確なフィールドタイプと検証要件を指定する
   - レイアウト環境設定およびユーザーエクスペリエンス要件を含める
   - 複雑なフォームをより小さな増分要求に分割します

2. **反復開発アプローチ**
   - 基本的なフォーム構造から開始
   - フィールドと機能を増分的に追加する
   - 続行する前に各追加をテスト
   - 単一の複雑なリクエストではなく、会話を通じて調整

3. **プロンプトの例**

   以下の代わりに使用します。

        顧客用フォームの作成 
   
   使用方法:

        顧客連絡先フォームの作成：
        - フルネーム （必須テキストフィールド） 
        - メールアドレス （検証で必要） 
        – 電話番号（オプション、フォーマット済み） 
       - メッセージ （必須 textarea、最大 500 文字） 
        - メール通知に送信 
   
### レイアウトとスタイルの問題

**症状：**

- モバイルデバイスでフォームが壊れているように見える
- 間隔または配置に一貫性がない
- フィールドが正しく表示されない
- 視覚階層不良

**解決策：**

1. **モバイルの応答性**
   - モバイル固有の最適化をリクエスト：「このフォームをモバイルに対応させる」
   - レスポンシブデザイン要件の指定
   - 実際のモバイルデバイスでのテスト
   - モバイルに単一列レイアウトを使用

2. **レイアウトの改善**
   - レイアウト要件を具体的に指定します。「アドレスフィールドを 2 列に配置します」
   - 特定のスタイル設定を要求：「プロフェッショナルな色を使用してタイポグラフィをきれいに」
   - 必要な間隔と位置合わせを指定する
   - アクセシビリティコンプライアンスの依頼

3. **ブランドの一貫性**
   - フォーム作成前にブランドガイドラインを準備
   - リクエストに特定のカラーコードとフォントを含める
   - すべてのフォームで一貫したスタイル設定を使用する
   - 再利用するブランドテンプレートの作成

### 条件付きロジックの問題

**症状：**

- ルールが期待どおりにトリガーされない
- フィールドの表示/非表示が正しくない
- 検証ロジックが機能しない
- 複雑なビジネスルールの失敗

**解決策：**

1. **ルールの簡略化**
   - 複雑なルールをより小さく、よりシンプルな条件に分割
   - 結合する前に各ルールを個別にテスト
   - 明確で具体的な条件を使用：「@maritalStatus が「既婚」に等しい場合に@spouseInfo を表示」
   - ネストされたロジックや過度に複雑なロジックを最初から避けます。

2. **ルールのテストとデバッグ**
   - 考えられるすべてのユーザーパスとシナリオのテスト
   - 条件でのフィールド名と値の検証
   - ルール条件で大文字と小文字を区別するかどうかを確認します
   - デバッグモードを使用したルール実行のトレース

3. **ビジネスロジックの実装**
   - 実装前にビジネス要件を明確に文書化
   - ルールを増分的に実装し、各手順をテストする
   - ルールがトリガーされた際の、明確なユーザーフィードバックの提供
   - エッジケースと例外シナリオの処理

## ファイルの読み込みと変換の問題

### PDF読み込みエラー

**症状：**

- PDF ファイルがアップロードまたは処理されない
- 変換後のフォームにフィールドまたはコンテンツが見つからない
- PDFの変換中のエラーメッセージ
- PDFからのフィールド認識が低い

**解決策：**

1. **ファイル形式とサイズ**
   - PDFのファイルが 10 MB 未満であることを確認します
   - 高品質のテキストベースの PDF を使用（スキャン画像は使用しない）
   - PDFがパスワードで保護されていないか、暗号化されていないことを確認
   - テキストの抽出に失敗した場合は、PDFを画像形式に変換してみてください

2. **PDF品質の最適化**
   - 明確に定義されたフォームフィールドを使用した PDF の使用
   - 適切なコントラストと読みやすいテキストを確保する
   - 複雑なレイアウトや要素の重複を避ける
   - コンバージョンリクエストに追加のコンテキストを提供

3. **コンバージョンの強化**
   - 期待されるフォーム構造の詳細
   - フィールドタイプと検証要件の指定
   - リクエスト固有の改善点：「モバイルの応答性と検証を追加する」
   - 変換後のフォームを手動でレビューおよび調整

### 画像とスクリーンショットの変換に関する問題

**症状：**

- 画像からの視野認識の不良
- フィールドのタイプまたはレイアウトが正しくありません
- フォーム要素がありません
- コンバージョンエラーまたはタイムアウト

**解決策：**

1. **必要な画像品質**
   - 高解像度の画像を使用（300 DPI 以上）
   - 良好な照明とコントラストを確保
   - 影、グレア、または歪みを避ける
   - 画像を切り抜いてフォームコンテンツのみにフォーカス

2. **最適な画像形式**
   - PNG またはJPG形式を使用すると効果的です
   - GIFや低画質の圧縮画像は避ける
   - 画像内のテキストをはっきりと読みやすくします。
   - 必要に応じて、別の画像の向きを試す

3. **コンバージョンガイダンス**
   - フォーム構造の詳細な説明を提供
   - フィールドタイプと要件を明示的に指定する
   - 変換中のリクエスト固有の機能強化
   - 変換後に手動で調整する準備が整っている

## 統合と送信に関する問題

### フォーム送信エラー

**症状：**

- Formsが正常に送信されない
- 送信中のエラーメッセージ
- データが目的の宛先に到達していない
- 送信中のタイムアウトエラー

**解決策：**

1. **送信アクションの設定**
   - 送信アクションが正しく設定されていることを確認します。
   - API エンドポイントと認証資格情報を確認します
   - 最初に簡単なメール送信でテスト
   - データ形式要件の検証

2. **ネットワークと接続性**
   - インターネット接続とネットワークの安定性の確認
   - ファイアウォール設定でフォーム送信が許可されていることを確認する
   - 異なるネットワーク接続からのテスト
   - 企業のプロキシまたはセキュリティの制限を確認します

3. **データ検証の問題**
   - すべての必須フィールドに入力する必要があります
   - データ形式が API 要件に一致することを確認します。
   - 特殊文字またはエンコーディングの問題を確認する
   - 最初に最小限のデータセットでテスト

### API 統合に関する問題

**症状：**

- REST API エンドポイントが応答しない
- 認証エラー
- データ形式の不一致
- 統合のタイムアウトまたはエラー

**解決策：**

1. **API 設定の検証**
   - API エンドポイント URL が正しく、アクセス可能であることを確認します
   - 認証資格情報と権限の確認
   - Postmanなどのツールを使用して、API エンドポイントを個別にテストします。
   - API が正しいデータ形式（JSON、XML など）を受け入れていることを確認

2. **データマッピングの問題**
   - フォームフィールド名が API パラメーターの要件に一致することを確認します。
   - 欠落している可能性のある必須フィールドをチェックします
   - データタイプの互換性の確認（文字列、数値、日付）
   - サンプルデータを使用したテストによるマッピングの問題の特定

3. **エラー処理とデバッグ**
   - API 呼び出しの詳細なエラーログを有効にする
   - API 応答コードとエラーメッセージの確認
   - 一時的なエラーに対する再試行ロジックの実装
   - API が失敗した場合のユーザーに対するフォールバックオプションの提供

### メール統合に関する問題

**症状：**

- 確認メールが送信されない
- E メールがスパムフォルダーに送信される
- メールの形式が正しくない
- メールにフォームデータがない

**解決策：**

1. **メール設定**
   - メールアドレスの形式が正しいことを確認します
   - SMTP 設定と認証を確認
   - 最初にシンプルなメールアドレスでテスト
   - メールサーバーの権限とクォータの確認

2. **メール配信の最適化**
   - 適切なメールヘッダーと送信者情報を使用する
   - 件名行にスパムトリガーの言葉を含めない
   - 適切な購読解除メカニズムを含める
   - 様々なプロバイダーへのメール配信のテスト

3. **コンテンツと書式設定**
   - フォームデータがメール内で適切にフォーマットされていることを確認します
   - 特殊文字またはエンコーディングの問題を確認する
   - 様々なデータの組み合わせでメールテンプレートをテストします
   - メールコンテンツにアクセスして読み取れるようにする

## パフォーマンスと読み込みの問題

### フォームの読み込みが遅い

**症状：**

- Formsの最初の読み込みに時間がかかる
- ユーザーインタラクションの遅延
- フォーム操作中のタイムアウト
- モバイルデバイスのパフォーマンスが低い

**解決策：**

1. **フォームの最適化**
   - フィールドの数と複雑さを軽減
   - 重要でないセクションに対する遅延読み込みの実装
   - Web 配信用の画像とアセットの最適化
   - 不要な検証ルールまたはロジックの削除

2. **ブラウザーとデバイスの最適化**
   - ブラウザーキャッシュと一時ファイルのクリア
   - 不要なブラウザータブとアプリケーションを閉じる
   - 使用可能なデバイスのメモリとストレージを確認します
   - パフォーマンス比較のために別のブラウザーを試す

3. **ネットワークの最適化**
   - 異なるネットワーク接続でのテスト
   - ネットワークの輻輳や帯域幅の制限をチェックします。
   - 可能であれば WiFi ではなく有線接続を使用してください
   - ネットワークパフォーマンスの問題については、IT サポートにお問い合わせください

### 検証パフォーマンスの問題

**症状：**

- 検証応答が遅い
- エラーメッセージの遅延表示
- 検証中のフォームのフリーズ
- フィールド検証中のタイムアウトエラー

**解決策：**

1. **検証の最適化**
   - リアルタイム検証の頻度を削減
   - 検証呼び出しのデバウンスの実装
   - 複雑な検証ルールをシンプル化
   - 可能な場合はクライアントサイド検証を使用

2. **ルールの簡略化**
   - 複雑な検証を小さなルールに分割
   - 不要なクロスフィールド検証の削除
   - パフォーマンス向けの条件付きロジックの最適化
   - 必要に応じて検証結果をキャッシュする

3. **ユーザーエクスペリエンスの向上**
   - シンプルな検証のためのフィードバックを迅速に提供
   - 複雑なルールに対して、リアルタイムではなくプログレッシブ検証を使用する
   - 検証プロセス中に読み込み指標を表示
   - 検証がバックグラウンドで処理されている間、ユーザーが続行できるようにする

## 詳細なトラブルシューティング

### デバッグモードと診断

**デバッグ情報を有効にする**

これらのプロンプトを使用して、フォームの問題に関する詳細情報を取得します。

     デバッグモードを有効にして、フォーム送信とフィールド検証の問題を特定 
    
     フォームエラーを分析：検証ルール、API 応答、ユーザー入力パターンを確認 
    
     フォーム構造とフィールド設定に関する詳細情報を表示 

### エラー分析テクニック

**体系的なデバッグアプローチ**

1. **問題を特定する**
   - 最小限のフォーム設定でテスト
   - 複雑なフィーチャを一時的に削除する
   - 個々のコンポーネントを個別にテストする
   - 根本的な原因を特定するための排除プロセスの使用

2. **診断情報の収集**
   - ブラウザーコンソールでJavaScript エラーを確認します
   - ネットワークリクエストと応答の確認
   - 問題を再現するための正確な手順の文書化
   - スクリーンショットとエラーメッセージの収集

3. **テスト環境変数**
   - 様々なブラウザーやデバイスを試す
   - 様々なユーザーアカウントと権限でのテスト
   - 異なるネットワーク環境での検証
   - 作業用フォームや設定との比較

### ログの分析と監視

**ブラウザーコンソールのデバッグ**

1. ブラウザー開発者ツールを開く（F12）
2. 「コンソール」タブでJavaScript エラーを確認する
3. 失敗したリクエストの「ネットワーク」タブを確認する
4. 処理に時間がかかる場合の「パフォーマンスの監視」タブ

**一般的なエラーパターン**

- **CORS エラー**:API 統合におけるクロスオリジンリクエストの問題
- **認証エラー**：無効な資格情報または期限切れのトークン
- **検証エラー**：フィールド検証ルールの競合または構文エラー
- **ネットワークタイムアウト**：ネットワーク接続が遅い、または信頼できない
