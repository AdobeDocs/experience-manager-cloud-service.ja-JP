# ユニバーサルエディターを使用したレスポンシブFormsの作成

ユーザーは、デスクトップ、タブレット、スマートフォンなど様々なデバイス上のフォームにアクセスします。 レスポンシブフォームをデザインすると、デバイスに関係なく、すべてのユーザーに最適なエクスペリエンスが提供されます。 このガイドでは、ユニバーサルエディターを使用してフォームを設計、テストおよび最適化し、あらゆる画面サイズに対応する方法を説明します。

レスポンシブフォームの作成には、次の 2 つの主なアクティビティが含まれます。

- **レスポンシブテスト：** デバイスエミュレーターを使用して、様々な画面サイズでフォームをプレビューおよびテストします。
- **レスポンシブデザイン：** 様々なデバイスにシームレスに適応するレイアウトパターンを選択し、実装します。

**このガイドでは、次の方法を説明します。**

- デスクトップ、タブレット、モバイルデバイスでのフォームのテスト
- コンテンツに適したレイアウトパターンの選択
- レスポンシブデザインのベストプラクティスの適用
- レスポンシブフォームに関するよくある問題のトラブルシューティング
- モバイルパフォーマンス向けにフォームを最適化

## レスポンシブFormsが重要な理由

**ユーザーエクスペリエンスへの影響：**

- 60% を超えるユーザーがモバイルデバイス上のフォームにアクセスしています
- モバイルエクスペリエンスが低いと、放棄率が 67% 高くなります
- レスポンシブフォームにより、完了率を最大 25% 向上させることができます

**ビジネス上のメリット：**

- フォームの完了率の上昇
- ユーザー満足度の向上
- アクセシビリティコンプライアンスの強化
- 開発コストとメンテナンスコストの削減

>[!TIP]
>
> **モバイルファーストのアプローチ：** モバイルデバイスの設計を開始してから、より大きな画面の拡張を行います。 これにより、コア機能が最も制約の厳しい環境で機能するようになります。

## 第 1 部：デバイス間でのFormsのテスト

異なるデバイス上のフォームをテストすると、ユーザーが問題に遭遇する前に、レスポンシブな問題を特定して解決するのに役立ちます。 ユニバーサルエディターは、様々な画面サイズと向きをシミュレートするエミュレーターモードを提供します。

### Formsのテスト方法

**手順 1：デバイスエミュレーターを開く**

1. フォームをユニバーサルエディターで開きます。
2. ツールバーの ![ エミュレーターアイコン ](/help/edge/docs/forms/universal-editor/assets/emulator.png){height=2%,width=2%} **エミュレーター** アイコンをクリックします。
3. デバイスセレクターメニューが表示されます。

![ ユニバーサルエディターのレスポンシブテストインターフェイス ](/help/edge/docs/forms/universal-editor/assets/universal-editor-emulator.png)

**手順 2：テストするデバイスの選択**

- **デスクトップ** （1200px+幅）：デフォルトの編集ビュー
- **Tablet** （幅 768px-1199px）: Mediumのスクリーンテスト
- **モバイル** （幅 320px-767px）：小型画面のテスト
- **カスタム**：特定のデバイスの正確な寸法を指定します

**手順 3：デバイスの向きをテストする**

**スクリーンローテータ** アイコンを使用して、次の両方をテストします。

- **縦長モード**：標準のモバイル向き
- **横置きモード**：タブレットまたは電話の回転表示

### 医療機器テスト結果

各デバイスタイプには、一意のレスポンシブ動作が表示されます。

| **デバイスタイプ** | **画面の幅** | **確認対象** | **よくある問題** |
|-----------------|------------------|-------------------|-------------------|
| **デスクトップ** | 1200px+ | フルレイアウト、すべての機能を表示 | 過度の空白、過度に広いレイアウト |
| **タブレット** | 768px-1199px | コンポーネントの積み重ね、ナビゲーションの操作性 | サイズ設定が難しく、ターゲットに影響する問題が発生する |
| **モバイル** | 320px-767px | 1 列レイアウト、サムナビゲーション | 小さいテキスト、間隔の狭いボタン |
| **カスタム** | ユーザー定義 | デバイス固有の要件 | ブレークポイントエッジケース |

### ワークフローのテスト

**ニューFormsの場合：**

1. **デスクトップビューでの作成：** すべての機能を使って作業を開始します。
2. **タブレットでテスト：** 中程度の画面の適応を確認します。
3. **モバイルで検証：** 小さな画面での操作性を確保します。
4. **レスポンシブな問題の修正：** 必要に応じてレイアウトを調整します。
5. **すべてのデバイスを再テスト：** すべてのサイズで修正を確認します。

**既存のFormsの場合：**

1. **クイックモバイルチェック：** フォームは電話で機能しますか？
2. **問題領域の特定：** レイアウトと操作性の問題に注意してください。
3. **システマティックテスト：** 各デバイスのサイズを徹底的にテストします。
4. **ドキュメントのイシュー：** 修正が必要な事項を追跡します。
5. **修正の実装：** 問題に適切な方法で対処します。

## パート 2：レスポンシブレイアウトパターンの選択

レイアウトパターンは、様々な画面サイズに対するフォームコンテンツの適応方法を決定します。 適切なパターンにより、ユーザーエクスペリエンスとモバイルパフォーマンスの両方が向上します。

### レイアウトパターンの概要

| **レイアウトタイプ** | **次に最適** | **モバイルのパフォーマンス** | **複雑性** |
|---------------------|-------------------------------------------|------------------------|----------------|
| **パネルレイアウト** | 分類されたコンテンツ、ダッシュボードスタイルのフォーム | 良好 | 低 |
| **ウィザードレイアウト** | 複数のステップから成るプロセス、複雑なワークフロー | 素晴らしい | 中 |
| **アコーディオンレイアウト** | FAQ スタイルのコンテンツ、オプションセクション | 素晴らしい | 低 |

### クイック決定ガイド

**パネルレイアウトを使用する場合：**

- コンテンツは異なるカテゴリに分類されています
- ユーザーは一度に複数のセクションを表示する必要がある
- コンテンツは比較的シンプルです

**次の場合にウィザード レイアウトを使用する：**

- フォームには複数の論理ステップがあります
- 認知負荷を軽減する
- モバイルユーザーは主なオーディエンスです

**アコーディオンレイアウトは次の場合に使用します。**

- フォームには、オプションまたはセカンダリコンテンツが含まれています
- 省スペースも重要
- コンテンツは論理的にグループ化できます

### パネルレイアウト

**目的：** 関連するコンテンツを、同時に表示できる視覚的に異なるセクションに整理します。

![ パネルレイアウトの例 ](/help/edge/docs/forms/universal-editor/assets/panel-layout.png)

**レスポンシブ動作：**

- **デスクトップ （1200px+）:** パネルが左右に並べて表示されるか、グリッドに表示されます
- **タブレット （768px-1199px）:** パネルが間隔を置いて縦に積み重なります
- **モバイル （320px-767px）:** セクション区切りがクリアな単一列レイアウト

**実装手順：**

1. [ パネルコンポーネント ](https://experienceleague.adobe.com/ja/docs/experience-manager-core-components/using/adaptive-forms/adaptive-forms-components/panel) を使用します。
2. 各パネル内の関連フィールドをグループ化します。
3. 各セクションに明確な見出しを追加します。
4. パネル間に適切なスペースを確保します。

**ベストプラクティス：**

- デスクトップパネルの数は 3～4 に制限して、ユーザーが圧倒されないようにします。
- 各パネルに説明タイトルを使用します。
- 関連するフィールドを論理的にグループ化して、認知負荷を軽減します。
- タッチデバイスでパネルナビゲーションをテストします。

**ユースケースの例：**

- **応募：** 個人情報，教育，経験，参考資料
- **製品登録：** 基本情報、技術仕様、保証情報
- **調査Forms:** 人口統計、環境設定、フィードバック、連絡先

### ウィザードレイアウト

**目的：** 複雑なプロセスを段階的にガイドし、認知負荷の軽減と完了率の向上を実現します。

![ ウィザードレイアウトの例 ](/help/edge/docs/forms/universal-editor/assets/wizard-layout.png)

**レスポンシブ動作：**

- **すべてのデバイス：** 最適なモバイルエクスペリエンスを実現するために、1 回の操作でフォーカスを維持します。
- **ステップコンテンツ：** 各ステップ（積み重ねまたは並べて）内で適応します。
- **ナビゲーション：** 十分な間隔を持つタッチフレンドリーなボタン。
- **プログレスインジケーター：** 画面サイズに合わせて適切に拡大・縮小します。

**実装手順：**

1. [ ウィザードコンポーネント ](https://experienceleague.adobe.com/ja/docs/experience-manager-core-components/using/adaptive-forms/adaptive-forms-components/wizard) を使用します。
2. 複雑なフォームを論理的なステップに分割します（3～7 段階が最適です）。
3. ユーザーの向きの進捗状況インジケーターを含めます。
4. 明確なナビゲーションコントロール（次へ、戻る、保存）を提供します。

**モバイルの最適化：**

- ナビゲーションボタンにはラージタッチターゲット（最小 44 px）を使用します。
- 小さい画面でステップインジケーターが明確に表示されていることを確認します。
- 1 ステップあたりのフィールド数を制限して、スクロールを減らします。
- 自動保存を有効にして、データの損失を防ぎます。

**ベストプラクティス：**

- 論理的なステップの進行を保証します。各ステップは、前のステップに基づいて構築する必要があります。
- 明確なステップタイトルを使用して、ユーザーが期待通りの結果を得られるようにします。
- エラーを早期に検出するために、各ステップで入力を検証します。
- ユーザーが逆方向に移動して情報をレビューまたは編集できるようにする。

**ユースケースの例：**

- **保険金請求：証拠→人→確認→の** 件
- **アカウント設定：** 基本情報→環境設定→ セキュリティ→確認
- **受注処理：** 製品→出荷→支払→要約

### アコーディオンレイアウト

**目的：** コンテンツを折りたたみ可能なセクションに整理することでスペースを節約します。オプションの情報やセカンダリ情報に最適です。

![ アコーディオンレイアウトの例 ](/help/edge/docs/forms/universal-editor/assets/accordion-layout.png)

**レスポンシブ動作：**

- **優れたモバイルパフォーマンス：** 関連性の高いコンテンツのみが表示されます。
- **タッチ操作向けヘッダー：** セクションのタップや展開が簡単です。
- **スムーズ アニメーション：** インタラクションの視覚的なフィードバックを提供します。
- **省スペース：** すべてのデバイスでスクロールを最小限に抑えます。

**実装手順：**

1. [ アコーディオンコンポーネント ](https://experienceleague.adobe.com/ja/docs/experience-manager-core-components/using/adaptive-forms/adaptive-forms-components/accordion) を使用します。
2. 各セクションで関連するオプションコンテンツをグループ化します。
3. 説明セクションのヘッダーを使用します。
4. 適切なデフォルトのオープン/クローズ状態を設定します。

**モバイルのメリット：**

- 使用されていないセクションを折りたたんでスクロールを減らします。
- 自然な展開/折りたたみジェスチャーによるタッチフレンドリーな操作。
- 読み込みの高速化 – アクティブなコンテンツのみが表示されます。
- フォーカスの向上：ユーザーは必要なものだけを表示できます。

**ベストプラクティス：**

- セクションのヘッダーをクリアを使用すると、内容を把握してから展開できます。
- 関連するコンテンツを各セクション内で論理的にグループ化します。
- 必要に応じて、重要なセクションを展開して開始するように設定します。
- 拡張する内容をユーザーが決定するのに役立つ、簡単なセクションのプレビューを提供します。

**ユースケースの例：**

- **製品構成：** 基本→高度な→ アクセサリ→サポート
- **よくある質問Forms:** アカウント→請求→ テクニカル→ース一般
- **設定Forms:** Privacy → Notifications → Appearance → Advanced

## 第 3 部：レスポンシブデザインのベストプラクティス

### デバイスタイプ別のベストプラクティス

+++モバイルの最適化（320px-767px）

**必要不可欠な慣行：**

- すべてのコンテンツに 1 列レイアウトを使用します。
- 大きくてタッチ操作に適したボタン（高さ 44 px 以上）を提供します。
- 戻る/次へオプションをクリアして、ナビゲーションを簡素化します。
- 各セクション内でスクロールを最小限に抑えます。
- キーボードを表示する最初のフィールドにオートフォーカスします。

**フィールド固有のガイドライン：**

- **テキスト入力：** 十分なパディングを持つ全幅。
- **ドロップダウン：** タッチエクスペリエンスを向上させるために、ネイティブの選択要素を使用します。
- **日付ピッカー：** モバイルの互換性のために、ネイティブの日付入力を使用します。
- **ファイルのアップロード：** 大きくてクリアなアップロード領域を提供します。

+++

+++タブレットの最適化（768px-1199px）

**レイアウト方法：**

- 関連するフィールドには、2 列のレイアウトを使用します。
- 縦向きと横向きの両方をテストします。
- タッチ操作とマウス操作の両方をサポートします。
- 読みやすさを維持しながら、より大きなコンテンツ領域を提供します。

+++

+++デスクトップの最適化（1200px+）

**高度な機能：**

- 複数列のレイアウトを使用すると、スペースを効率的に使用できます。
- パワーユーザー向けのキーボードショートカットを提供します。
- インタラクティブフィードバックのホバー状態を実装します。
- 詳細なエラーメッセージを使用して詳細な検証を提供します。

+++

## 包括的なトラブルシューティング

### レイアウトの問題

+++モバイルのフォームレイアウトが壊れる

**一般的な原因：**

- 拡大縮小しない固定幅要素
- デスクトップファーストのレイアウト向けに設計された CSS
- コンテナがオーバーフローする画像またはコンテンツ

**ソリューション：**

- 画像とコンテナを画面サイズに合わせて拡大・縮小する。
- プログレッシブな強化機能を備えたモバイルファーストのデザインを使用します。
- デバイスエミュレーターと実際のデバイスの両方を使用してテストします。
- 固定寸法ではなく、柔軟なサイズ設定を使用します。

+++

+++タッチターゲットが小さすぎる

**一般的な原因：**

- 44px × 44px 未満のボタン
- インタラクティブな要素が近すぎる
- タッチ操作に対応するデフォルトを上書きするカスタム CSS

**ソリューション：**

- すべてのインタラクティブ要素が 44 ピクセル × 44 ピクセル以上であることを確認してください。
- ボタンとリンクの間の間隔を追加します。
- マウスだけでなく、実際の指でタッチ操作をテストします。
- タップしやすくするために、タッチターゲット領域を増やします。

+++

+++コンテンツオーバーフローの問題

**一般的な原因：**

- 長いテキストまたは折り返さないラベル
- 固定幅コンテナ
- 適切に拡大/縮小されない画像

**ソリューション：**

- 長いコンテンツのテキストの折り返しを有効にします。
- 適切に拡大・縮小できるレスポンシブな画像を使用します。
- コンテンツに適応した柔軟なレイアウトの実装。
- 様々なコンテンツの長さでテストします。

+++

### パフォーマンスの問題

+++モバイルでの読み込みが遅い

**一般的な原因：**

- モバイル用に最適化されていない大きな画像
- JavaScriptの過度の実行
- 一度に読み込むフォームフィールドが多すぎます

**ソリューション：**

- 異なる画面サイズに合わせて画像を最適化します。
- 重要でないコンテンツは、必要な場合にのみ読み込みます。
- テクニックを使用して、モバイル読み込みを高速化します。
- サードパーティのスクリプトおよびウィジェットを最小限に抑えます。

+++

### テストと検証の問題

+++エミュレーターと実際のデバイスの違い

**一般的な原因：**

- ブラウザー固有のレンダリングの違い
- タッチ操作とマウス操作の違い
- ネットワーク速度の変動

**ソリューション：**

- 可能な限り、実際のデバイスでテストします。
- エミュレーターテストに複数のブラウザーを使用します。
- テスト中に様々なネットワーク速度をシミュレートします。
- ターゲット環境で実際のユーザーと検証します。

+++

## レスポンシブFormsの成功指標

+++主要業績評価指標

**ユーザーエクスペリエンス指標：**
