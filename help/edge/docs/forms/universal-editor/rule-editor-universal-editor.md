# ユニバーサルエディターの動的Forms用のルールエディター

ルールエディターを使用すると、作成者は、コードを記述することなく、静的なフォームをレスポンシブでインテリジェントなエクスペリエンスに変換できます。 フィールドの条件付き表示、計算の実行、データの検証、フローのガイドおよびユーザー入力に応じたビジネスロジックの統合を行うことができます。

## 学習内容

このガイドを終了すると、以下が可能になります。

- ルールの仕組みと、様々なルールタイプを使用するタイミングについて
- ユニバーサルエディターでのルールエディターの有効化とルールエディターへのアクセス
- フィールドを動的に表示または非表示にする条件付きロジックの作成
- 自動計算とデータ検証の実装
- 複雑なビジネスルールのカスタム関数の作成
- パフォーマンス、メンテナンス性、UX にベストプラクティスを適用

## ルールエディターを使用する理由

- **条件付きロジック**：ノイズと認知負荷を軽減するために必要な場合にのみ、関連するフィールドを表示します。
- **動的計算**：ユーザー・タイプに応じて値（合計、レート、税金）を自動的に計算します。
- **データの検証**：リアルタイムのチェックとクリアメッセージを使用して、早期にエラーを防ぎます。
- **ガイド付きエクスペリエンス**：論理的な手順（ウィザード、ブランチ）でユーザーを導きます。
- **コードなしオーサリング**：ビジュアルインターフェイスを通じて強力な動作を設定します。

一般的なシナリオとしては、税金計算、ローンと保険料の見積もり、実施要件フロー、複数手順の申請、条件付き質問を含む調査などがあります。

## ルールの仕組み

ルールは、条件が満たされた場合に何を行うかを定義します。 ルールには概念的に次の 2 つの部分があります。

- **条件**:true または false と評価されるステートメント。
   - 例：「収入 > 50,000」、「カバレッジ = 「はい」、「フィールドが空です」
- **アクション**：条件が true の場合（およびオプションで false の場合）の動作。
   - 例：フィールドの表示/非表示、値の設定/消去、入力の検証、ボタンの有効化/無効化

+++ ルールの論理パターン

- **条件→アクション（When/Then）**

  ```text
  WHEN Gross Salary > 50000
  THEN Show "Additional Deduction"
  ```

  条件付き表示およびプログレッシブ開示に最適です。

- **アクション ←条件（If/Only If を設定）**

  ```text
  SET Taxable Income = Gross Salary - Deductions
  IF Deductions are applicable
  ```

  計算やデータ変換に最適です。

- **If → Then → Else （代替アクション）**

  ```text
  IF Income > 50000
  THEN Show "High Income" fields
  ELSE Show "Standard Income" fields
  ```

  分岐ロジックおよび相互に排他的なフローに最適です。

+++

+++ 実際の例

- **条件**:「総給与が 5 万ドルを超える」
- **プライマリ操作**: 「追加控除」を表示します
- **代替アクション**:「追加控除」を非表示にします
- **結果**：ユーザーには、適用されるフィールドのみが表示されます

## 使用可能なルールタイプ

目的に最適なルールタイプを選択します。

+++ 条件付きロジック

- **When**：複雑な条件動作のプライマリルール（条件→動作±Else）
- **非表示/表示**：条件に基づいて表示を制御します（プログレッシブ表示）。
- **有効/無効**：フィールドがインタラクティブかどうかを制御します（例えば、必須フィールドが有効になるまで「送信」を無効にします）

+++

+++ データ操作

- **値を設定**：値（日付、合計、コピーなど）を自動入力します
- **値をクリア**：条件が変更された場合、データを削除します
- **形式**：格納された値を変更せずに表示形式（通貨、電話、日付）を変換します

+++

+++ 検証

- **検証**：クロスフィールドチェックやビジネスルールを含むカスタム検証ロジック

+++

+++ 計算

- **数式**：値をリアルタイムで計算（合計、税、比率）

+++

+++ ユーザーインターフェイス

- **フォーカスの設定**：特定のフィールドにフォーカスを移動する（慎重に使用する）
- **プロパティを設定**：コンポーネントプロパティ（プレースホルダー、オプションなど）を動的に変更します

+++

+++ フォームコントロール

- **フォームの送信**：プログラムによってフォームを送信します（検証に合格した後のみ）
- **フォームをリセット**：クリアして初期状態にリセット（使用前に確認してください）
- **フォームを保存**：後で使用するためにドラフトとして保存（長いフォーム、複数セッション）

+++

+++ 詳細

- **サービスを呼び出し**：外部 API/サービスを呼び出します（読み込みとエラーを処理）
- **インスタンスの追加/削除**：繰り返し可能なセクション（依存、アドレスなど）を管理します。
- **移動先**：他のフォームまたはページにルーティングします（ナビゲーション前にデータを保持します）
- **パネル間を移動**：ウィザードの手順のナビゲーションとスキップを制御する
- **イベントのディスパッチ**：統合または分析用のカスタムイベントをトリガーします

+++

## ステップバイステップのチュートリアル：スマート税計算ツールの作成

+++ チュートリアルの概要

この例では、条件付き表示と自動計算を示します。

![ フォームフィールド表示の「When-Then」ロジックを含む条件付きルールの作成を示すルールエディターインターフェイスのスクリーンショット ](/help/edge/docs/forms/assets/rule-editor-1.png)
図：インテリジェントな条件付きフィールドを備えた税金計算フォーム

次のようなフォームを作成します。

1. 関連するフィールドを表示することで、ユーザー入力に適応する
2. リアルタイムで値を計算します
3. データを検証して精度を向上

+++

+++ フォーム構造

| フィールド名 | タイプ | 目的 | 動作 |
|-------------------------|---------------|--------------------------------|-----------------------------------------|
| 総給与 | 数値入力 | ユーザーの年収 | トリガーの条件付きロジック |
| 追加控除 | 数値入力 | 追加控除（適格な場合） | 給与 > $50,000 の場合にのみ表示されます |
| 課税所得 | 数値入力 | 計算された値 | 読み取り専用、変更時に更新 |
| 未払税 | 数値入力 | 計算された値 | 読み取り専用、一定の割合で計算 |

+++

+++ ビジネスロジック

- **ルール 1：条件付き表示**

  ```text
  WHEN Gross Salary > 50,000
  THEN Show "Additional Deduction"
  ELSE Hide "Additional Deduction"
  ```

- **規則 2：課税所得の計算**

  ```text
  SET Taxable Income = Gross Salary - Additional Deduction
  (Only when Additional Deduction applies)
  ```

- **規則 3：未払税の計算**

  ```text
  SET Tax Payable = Taxable Income × 10%
  (Simplified flat rate)
  ```

+++

+++ 手順 1：基盤フォームの作成

**目的**：すべてのフィールドと初期設定を含むベースフォームを作成します。

1. **ユニバーサルエディターを開く**:
   - AEM Sites コンソールに移動して、ページを選択し、「**編集**」をクリックします
   - [ ユニバーサルエディター ](https://experienceleague.adobe.com/docs/experience-manager-cloud-service/content/implementing/developing/universal-editor/introduction.html) が正しく設定されていることを確認します

2. **フォームコンポーネントをこの順序で追加する**:
   - タイトル（H2）:「税金計算フォーム」
   - 数値入力：「総給与」（必須：「Yes」、プレースホルダ：「年給を入力」）
   - 数値入力：「追加控除」（必須：「No」、プレースホルダ：「追加控除の入力」）
   - 数値入力：「課税所得」（読み取り専用：はい）
   - 数値入力：「税金買掛金」（読み取り専用：Yes）
   - 送信ボタン：「税金の計算」

3. **最初のフィールドのプロパティを設定する**:
   - 「追加控除」を非表示（設定が表示：プロパティパネルで「いいえ」）
   - 「課税対象所得」および「税額未払」を読み取り専用に設定する：はい

![ 総給与、婚姻ステータス、扶養子の入力フィールドを含む税金計算フォームのスクリーンショット。ルールが適用される前のフォーム構造を示しています ](/help/edge/docs/forms/assets/rule-editor2.png)
図：基本コンポーネントが設定された初期フォーム構造

**チェックポイント**:「追加控除」が非表示で、計算フィールドが読み取り専用のすべての必須フィールドを含むフォームが必要です。

+++

+++ 手順 2：条件付き表示ルールの追加

**目標**：総給与が$50,000 を超える場合にのみ、「追加控除」フィールドを表示します。

1. **「総給与」フィールドを選択**、「ルールエディタ」アイコン ![edit-rules](/help/forms/assets/edit-rules-icon.svg) をクリックします
2. **新しいルールの作成**:
   - 「**作成**」をクリックします
   - ルールタイプを「Set Value Of」から「**When」** に変更します
3. **条件を設定します**。
   - ドロップダウンから「**次より大きい」** を選択します
   - 「番号」フィールドに「`50000`」と入力します
4. **Then アクションを設定します**。
   - アクション **選択ドロップダウンから「表示」** を選択します
   - フォームオブジェクトから **追加控除」** フィールドをドラッグまたは選択します
5. **Else アクションを追加します**。
   - 「Add Else Section **をクリックします**
   - アクション **選択ドロップダウンから「非表示」** を選択します
   - 「追加控除 **フィールドを選択** ます
6. **ルールを保存**:「完了 **をクリックします**。

>[!NOTE]
>
> 代替アプローチ：「総給与」に関する「When」ルールではなく、「追加控除」フィールドに直接表示/非表示ルールを作成することで、同じ結果を得ることができます。

+++

+++ 手順 3：計算ルールの追加

**目標**：ユーザー入力に基づいて「課税所得」と「支払税額」を自動計算します。

**課税所得計算の構成**:

1. **「課税所得」フィールドを選択** て、ルールエディターを開きます
2. **数式を作成**:
   - **作成** をクリック→、「数式 **を選択**
   - ビルド式：**総給与−追加控除**
   - 「総給与」を最初のフィールドにドラッグ
   - 「マイナス **演算子を選択** ます
   - 「追加控除」を 2 番目のフィールドにドラッグします
3. **保存**:「完了 **をクリックします**

**税金買掛/未払金計算の構成**:

1. **「支払税」フィールドを選択** て、ルールエディターを開きます
2. **数式を作成**:
   - **作成** をクリック→、「数式 **を選択**
   - ビルド式：**課税所得× 10 ÷ 100**
   - 「課税対象所得」を最初のフィールドにドラッグ
   - **&quot;乗算&quot;** 演算子を選択します
   - `10` を数字として入力
   - 「式 **拡張」をクリックし** す。
   - **&quot;除算&quot;** 演算子を選択します
   - `100` を数字として入力
3. **保存**:「完了 **をクリックします**

+++

+++ 手順 4：フォームをテストする

**フロー全体をテストして実装を検証します**。

1. **フォームをプレビューする**：ユニバーサルエディターのプレビューモードをクリックします
2. **条件ロジックのテスト**:
   - 総給与= `30000` → 「追加控除」は非表示のままにします
   - 総給与= `60000`→「追加控除」を入力します
3. **計算のテスト**:
   - 総給与= `60000` の場合は、追加控除= `5000` と入力します
   - 課税所得の検証= `55000` （60000 - 5000）
   - 支払税の確認= `5500` （55000 × 10%）

![ フォームをプレビューする ](/help/edge/docs/forms/assets/rule-editor-form.png)
図：条件付きフィールドと自動計算を使用した完了済みの税金計算

**成功条件**：ユーザーが入力したときに、フォームでフィールドの動的な表示/非表示および値のリアルタイム計算が行われます。

## 詳細：カスタム関数

組み込み機能を超える複雑なビジネスロジックの場合は、ルールエディターとシームレスに統合されるカスタム JavaScript関数を作成できます。

+++ カスタム関数を使用するタイミング

**カスタム関数の理想的なシナリオ**:

- **複雑な計算**：数式規則では表現しにくい多段階の計算
- **ビジネス固有の検証**：組織や業界に固有のカスタム検証ロジック
- **データ変換**：形式変換、文字列操作またはデータ解析
- **外部統合**：内部 API またはサードパーティのサービスの呼び出し（制限あり）

**カスタム関数のメリット**:

- **再利用性**：一度書き込むと、複数のフォームやルールで使用できる
- **メンテナンス性**：更新とデバッグが容易な、一元化されたロジック
- **パフォーマンス**：複雑なルールチェーンと比較した、JavaScriptの実行の最適化
- **柔軟性**：標準ルールで対応できないエッジケースや複雑なシナリオを処理します

+++

+++ カスタム関数の作成と実装

**ファイルの場所**：すべてのカスタム関数は、Edge Delivery Services プロジェクトの `/blocks/form/functions.js` で定義する必要があります。

**開発ワークフロー**:

1. **関数のデザイン**
   - 説明的でアクション指向の関数名を使用する
   - 明確なパラメータータイプと戻り値の定義
   - エッジケースと無効な入力を適切に処理します

2. **実装**
   - JavaScriptにコメントを書いて明確に書く
   - 入力検証とエラー処理を含める
   - 統合前に関数を個別にテスト

3. **ドキュメント**
   - 包括的な JSDoc コメントの追加
   - 使用例とパラメーターの説明を含める
   - 制限や依存関係を文書化する

4. **デプロイメント**
   - 名前付きエクスポートを使用した関数のエクスポート
   - プロジェクトリポジトリへのデプロイ
   - テスト前にビルド完了を確認

**実装例**:

```javascript
/**
 * Concatenates first and last name with proper formatting
 * @name getFullName
 * @description Combines first and last name, handles edge cases like missing values
 * @param {string} firstName - The person's first name
 * @param {string} lastName - The person's last name  
 * @returns {string} Formatted full name or empty string if both inputs are invalid
 */
function getFullName(firstName, lastName) {
  // Handle null, undefined, or empty string inputs
  const first = (firstName || '').toString().trim();
  const last = (lastName || '').toString().trim();
  
  return `${first} ${last}`.trim();
}

/**
 * Calculates the number of days between two dates
 * @name days
 * @description Computes absolute difference in days, handles various date input formats
 * @param {Date|string} endDate - End date (Date object or ISO string)
 * @param {Date|string} startDate - Start date (Date object or ISO string)
 * @returns {number} Number of days between dates, 0 if inputs are invalid
 */
function days(endDate, startDate) {
  // Convert string inputs to Date objects
  const start = typeof startDate === 'string' ? new Date(startDate) : startDate;
  const end = typeof endDate === 'string' ? new Date(endDate) : endDate;

  // Validate date objects
  if (Number.isNaN(start.getTime()) || Number.isNaN(end.getTime())) {
    return 0;
  }

  // Calculate absolute difference in milliseconds, then convert to days
  const diffInMs = Math.abs(end.getTime() - start.getTime());
  return Math.floor(diffInMs / (1000 * 60 * 60 * 24));
}

// Export functions for use in Rule Editor
export { getFullName, days };
```

## ルール開発のベストプラクティス


+++ パフォーマンスの最適化

- ルールの複雑さを最小限に抑え、大きなロジックを小さな集中したルールに分割
- 頻度別にルールを並べ替える（最も一般的には最初）
- コンポーネントごとのルールセットの管理を維持
- ロジックを複製するよりも、再利用可能なカスタム関数を優先します

+++

+++ ユーザーエクスペリエンス

- 明確な検証とインラインフィードバックの提供
- 視覚的に不快な変更を行わないでください。show/hide は慎重に使用してください。
- デバイスとレイアウトをまたいだテスト

+++

+++ 開発ハイジーン

- エッジケースと既知の値を使用したテスト
- ブラウザー間での検証
- 仕組みだけでなく、複雑なルールの背後にある意図を文書化する
- 大きなフォームのルール在庫を管理します
- コンポーネントとルールには一貫した名前を使用する
- 非実稼動環境でのカスタム関数およびテストのバージョン設定

## よくある問題のトラブルシューティング


+++ ルールがトリガーされない

- コンポーネント名と参照の確認
- 実行順序を確認（上から下）
- 既知の値で条件を検証
- ブラウザーコンソールでブロッキングエラーを検査

+++

+++ 間違った動作

- 演算子とグループ化（AND/OR）を確認する
- 式フラグメントを個別にテスト
- データタイプの確認（数値と文字列）

+++

+++ パフォーマンスの問題

- 深くネストされた条件の簡略化
- プロファイルカスタム関数
- ルール内の外部呼び出しを最小化
- 特定のセレクターと参照の使用

+++

+++ カスタム関数の問題

- ファイル パスの確認：`/blocks/form/functions.js`
- 名前付きエクスポートが正しいことを確認します
- ビルドに変更が含まれていることを確認します
- デプロイ後のブラウザーキャッシュのクリア
- パラメータータイプとエラー処理を検証

+++

+++ ユニバーサルエディターの統合

- ルールエディター拡張機能が有効になっていることを確認します
- サポートされているコンポーネントを選択
- サポートされているブラウザーを使用する（Chrome、Firefox、Safari）
- 必要な権限があることを確認します

## 概要

ユニバーサルエディターのルールエディターは、静的フォームを、リアルタイムのユーザー入力に適応するインテリジェントでレスポンシブなエクスペリエンスに変換します。 条件付きロジック、自動計算、カスタムビジネスルールを活用すると、アプリケーションコードを記述しなくても高度なフォームワークフローを作成できます。

**学習した主な機能**:

- **条件付きロジック**：ユーザー入力に基づいてフィールドの表示/非表示を切り替え、フォーカスされた適切なエクスペリエンスを作成します
- **動的計算**：ユーザーがフォームを操作する際に、値（税金、合計、料金）を自動的に計算します
- **データ検証**：明確で実用的なフィードバックメッセージを使用してリアルタイム検証を実装します
- **カスタム関数**:JavaScriptの機能を拡張して、複雑なビジネスロジックと統合を実現します
- **パフォーマンスの最適化**：維持可能で効率的なルール開発にベストプラクティスを適用します

**配信された値**:

- **ユーザーエクスペリエンスの向上**：プログレッシブディスクロージャーとインテリジェントフォームフローにより、認知負荷を軽減
- **エラーの削減**：リアルタイム検証とガイド付き入力により、無効な送信を防ぎます
- **効率の向上**：計算とデータ入力を自動化して、ユーザーの作業を最小限に抑える
- **保守可能なソリューション**：組織全体で規模を拡張する、再利用可能で詳細にドキュメント化されたルールを作成します

**ビジネスへの影響**:

Formsは、データ収集、リードの選定、ユーザーエンゲージメントのための強力なツールになります。 ルールエディターを使用すると、技術に詳しくない作成者でも高度なビジネスロジックを実装して、開発コストを削減しながら、フォームの完了率とデータ品質を向上させることができます。
