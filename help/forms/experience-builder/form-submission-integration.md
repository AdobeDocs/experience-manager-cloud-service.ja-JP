# フォームの提出と統合

>[!NOTE]
>
> Forms Experience Builderは、早期アクセスプログラムで利用できます。 開始する前に、アクセスをリクエストし、許可されていることを確認してください。

Forms Experience Builderなら、フォームを外部システム、API、ビジネスワークフローと連携させる強力な統合機能を利用できます。 このガイドでは、フォーム送信を設定し、様々な統合シナリオを設定する方法について説明します。

## 送信設定オプション

### 電子メール送信

電子メールで送信するフォームを設定します。

**基本的なメール設定：**

- 受信者のメールアドレスを設定
- メールテンプレートの設定
- CCおよびBCC受信者の追加
- メール通知の設定

**高度なメール機能：**

- 動的な受信者の選択
- フォームデータを使用したメールテンプレート
- 添付ファイルの処理
- 電子メールの配信確認

### REST APIとの統合

フォームを外部APIとサービスに接続します。

**API エンドポイント設定：**

- REST API URLの設定
- 認証方法の設定
- リクエストヘッダーとパラメーターの設定
- 応答データの処理

**データマッピング：**

- フォームフィールドをAPI パラメーターにマッピングする
- データ形式の変換
- ネストされたJSON構造の処理
- エラー応答の管理

### クラウドストレージの統合

フォーム送信をクラウドストレージサービスに保存する：

**サポートされているプラットフォーム：**

- Microsoft Azure Blob Storage
- Amazon S3
- Google クラウドストレージ
- SharePoint Online

**設定オプション：**

- ストレージ資格情報の設定
- フォルダー構造の設定
- ファイル命名規則の設定
- アクセス権限の管理

### ワークフロー統合

フォームをビジネスプロセスワークフローに接続します。

**Microsoft Power Automate:**

- フォーム送信時のトリガーワークフロー
- フォームデータをワークフローステップに渡す
- ワークフロー応答の処理
- 承認プロセスの管理

**Adobe ワークフロー：**

- AEM Workflowとの統合
- 承認チェーンの設定
- 通知ステップの設定
- ドキュメント処理の管理

## フォーム送信の設定

### 手順1：送信設定へのアクセス

1. Forms Experience Builderでフォームを開く
2. 送信設定に移動します
3. 「フォーム送信の設定」を選択
4. 統合タイプの選択

### 手順2：メール送信の設定

**基本的なメール設定：**

    hr@company.comへのメール送信を次の条件で設定する：
     – 件名：「新入社員アプリケーション」 
     – メール本文にフォームデータを含める
     – 申込者に確認を送信

**高度なメール設定：**

    動的な電子メールのルーティングを設定：
     – 部署が「IT」に等しい場合はit-hr@company.com
     – 部署が「Sales」に等しい場合はsales-hr@company.com
    - デフォルトはhr@company.com
に送信

### 手順3:API統合の設定

**REST API設定：**

     フォームデータをREST エンドポイントに送信：
    - URL: https://api.company.com/forms/submit
     – 方法：POST
     – 認証：ベアラートークン 
    - コンテンツの種類：application/json

**データマッピングの例：**

     フォームフィールドをAPI:
    - firstName -> user.first_name
    - lastName -> user.last_name
    - email -> user.email_address
    - department -> user.department_id
にマッピングします

### 手順4：クラウドストレージの設定

**Azure Blob Storageの設定：**

    Azureにフォームの送信を保存：
    - コンテナ：form-submissions
    - フォルダー：/{year}/{month}/{day}/
    - ファイル形式：添付ファイルを含むJSON
    - アクセスレベル：プライベート 

## 統合の例

### 顧客フィードバックフォーム

**送信設定：**

- サポートチームへのメール通知
- API経由でCRM システムにデータを保存
- サポートチケットの自動作成
- お客様に確認メールを送信する

**実装：**
お客様からのフィードバックフォームの送信先：
1. フォームの詳細が記載されたメール <support@company.com>
2. CRM APIにPOSTして顧客レコードを作成
3. トリガーサポートチケット作成ワークフロー
4. お客様に感謝メールを送信する

### 従業員オンボーディングフォーム

**送信設定：**

- 新入社員情報を人事部にメールで送信
- SharePointでのドキュメントの保存
- トリガーオンボーディングワークフロー
- 様々なシステムでのユーザーアカウントの作成

**実装：**
従業員オンボーディングのプロセス：
1. 従業員の詳細を記載したメール <hr@company.com>
2. SharePointの従業員フォルダーへのドキュメントのアップロード
3. Power Automateでオンボーディングワークフローを開始する
4. 人事システムや電子メールなどのツールでアカウントを作成する

### リードジェネレーションフォーム

**送信設定：**

- MA基盤にリードデータを保存する
- 営業部門に通知を送る
- CRM システムへのリードの追加
- トリガーフォローアップメールの順序

**実装：**
リードジェネレーションのプロセス：
1. MARKETO APIへのリードデータの投稿
2. Adobe Salesforceでリードレコードを作成
3. 営業部門へのメールでリードの詳細を確認
4. 自動メールナーチャリングシーケンスを開始

## 高度な統合シナリオ

### マルチステップフォーム処理

**複雑なワークフロー統合：**

- フォームデータを外部システムと検証する
- 支払いゲートウェイを通じて支払いを処理する
- 文書と契約書の生成
- 複数の関係者に通知を送信する

### リアルタイムデータ検証

**API ベースの検証：**

- 会社のディレクトリに対する電子メールアドレスの検証
- 在庫システムにおける製品の在庫状況を確認する
- CRMの顧客情報の確認
- 支払い情報を検証

### 条件付き送信ルーティング

**フォームデータに基づく動的ルーティング：**

- 問い合わせタイプに基づいて異なる部門にルーティング
- 顧客層にもとづいて異なるシステムに送信します
- フォームの入力状況に応じて異なる処理をする
- 地域ごとに異なるビジネスルールを扱う

## セキュリティとコンプライアンス

### データ保護

**暗号化とセキュリティ：**

- 転送中の機密データの暗号化
- 安全なAPI認証情報とトークン
- 適切なアクセス制御の実装
- データ保持ポリシーに従う

### コンプライアンス要件

**GDPRとプライバシー：**

- 同意管理の導入
- データ書き出し機能
- データ削除要求を有効にする
- 監査記録の管理

**業界標準：**

- ヘルスケアフォームのHIPAA準拠
- 決済処理のためのPCI DSS
- 金融フォームのSOX準拠
- 業界固有の規制

## テストと検証

### 提出テスト

**テストシナリオ：**

- メールの配信とフォーマットを確認する
- API接続とデータマッピングのテスト
- クラウドストレージのアップロードの検証
- ワークフロートリガー機能の確認

**エラー処理：**

- ネットワーク障害シナリオのテスト
- エラーメッセージ表示の検証
- 再試行メカニズムの確認
- フォールバックオプションの確認

### パフォーマンスの最適化

**最適化戦略：**

- 非同期処理の実装
- 一括データにバッチ操作を使用する
- API呼び出し頻度の最適化
- 頻繁にアクセスされるデータのキャッシュ

### 一般的な問題

**メール配信の問題：**

- SMTP サーバー設定の確認
- 受信者の電子メールアドレスの確認
- スパムフィルター設定の確認
- メールテンプレートの書式設定をテストする

**API統合の問題：**

- API エンドポイント URLの検証
- 認証情報を確認
- リクエストの形式とヘッダーを検証する
- API レスポンス処理のレビュー

**ストレージ統合の問題：**

- ストレージ資格情報の確認
- フォルダーの権限を確認
- ファイルのアップロード制限の確認
- ネットワーク接続のテスト
