# アダプティブForms：コアコンポーネントとEdge Delivery Services Formsと基盤コンポーネントの比較

Adobe Experience Manager（AEM）Formsは、データキャプチャエクスペリエンスを構築するための3つの異なるアプローチを提供します。コアコンポーネントに基づくアダプティブForms、Edge Delivery Services Forms、基盤コンポーネントに基づくアダプティブFormsです。 各アプローチは、アーキテクチャ、レンダリングモデル、ターゲットユースケースが異なります。 この記事では、ソリューションアーキテクト、開発者、AEMのお客様が要件に適したアプローチを選択するのに役立つ技術比較について説明します。

## 概要

3つのフォームはすべて、ユーザーデータを取得し、バックエンドシステムと統合することを目的としています。 ただし、基本となるアーキテクチャ、フォームのレンダリング方法、配信方法、サポートする機能などが異なります。

| アプローチ | ステータス | プライマリの使用例 |
|----------|--------|------------------|
| **コアコンポーネント** | 新しいフォーム向け | 柔軟性の高い公開オプションを備えた、最先端のスケーラブルフォームには、AEMオーサリングが必要 |
| **Edge Delivery Services Forms** | パフォーマンスが重要なサイト向け | 高速なデプロイメントでエッジから配信される高性能フォーム |
| **基盤コンポーネント** | メンテナンスモード | 従来の機能サポートを必要とする既存フォーム |

## コアコンポーネントに基づくアダプティブForms

### 定義

アダプティブ Forms コアコンポーネントは、Adobe Experience Manager WCM コアコンポーネントの基盤の上に構築された、BEM準拠の30個のオープンソースコンポーネントのセットです。 これは、Adobeが推奨する新しいアダプティブFormsの作成アプローチを表しており、最新のアーキテクチャ、パフォーマンスの向上、自動ヘッドレスフォーム生成を提供します。

### アーキテクチャ

コアコンポーネントでは、標準化されたモジュール式のコンポーネントアーキテクチャを使用しています。

- **コンポーネント基盤**: AEM WCM コアコンポーネントに基づいて構築
- **スタイル化手法**: BEM （ブロック要素修飾子） CSS規則
- **コンテンツストレージ**：構造化コンテンツノードを含むJCR リポジトリ
- **レンダリング**: AEMでのサーバーサイドレンダリングと、オプションのクライアントサイドヘッドレスレンダリング
- **Source**: オープンソース （[GitHub](https://github.com/adobe/aem-core-forms-components)で利用可能）

### レンダリングモデル

コアコンポーネントは、複数のレンダリングモデルをサポートしています。

1. **サーバーサイドレンダリング（SSR）**: FormsはAEM サーバー上でレンダリングされ、HTML全体をブラウザーに配信します
2. **ヘッドレスレンダリング**: Formsは、React、Angular、またはその他のフレームワークでのクライアントサイドレンダリング用のAPIを介してJSON表現を公開します
3. **ハイブリッドレンダリング**：最適なパフォーマンスを実現するためのサーバーとクライアントレンダリングの組み合わせ

### オプションを公開中

- AEM パブリッシュインスタンス
- Edge Delivery Services（設定済み）
- カスタムフロントエンドアプリケーション用のヘッドレス API

### 主な機能

| カテゴリ | 機能 |
|----------|-------------|
| **コンポーネント** | テキストボックス、数値ボックス、日付選択、ドロップダウン、チェックボックスグループ、ラジオボタン、ファイル添付、ウィザード、アコーディオン、水平/垂直タブを含む30の標準化されたコンポーネント |
| **フォームモデル** | JSON スキーマ（v4および2020-12）、フォームデータモデル（FDM）、XDP/XFA テンプレート（限定） |
| **ルールエンジン** | 条件付きロジック、検証、サービス呼び出し、カスタム関数を備えたビジュアルルールエディター |
| **送信アクション** | REST エンドポイント、電子メール、AEM ワークフロー、SharePoint、OneDrive、Azure Blob Storage、Power Automate、Workfront Fusion、フォームデータモデル |
| **事前入力** | フォームデータモデル事前入力サービス、カスタム事前入力サービス |
| **CAPTCHA** | reCAPTCHA、Captcha、Turnstile |
| **レコードのドキュメント** | カスタムテンプレートまたはOOTB テンプレートを使用したPDFの生成 |
| **アクセシビリティ** | WCAG準拠、ARIA ラベル、キーボードナビゲーション、スクリーンリーダーのサポート |
| **ローカライゼーション** | AEMの翻訳ワークフローによる多言語サポート |
| **バージョン管理** | AEM Sitesから継承されたコンテンツのバージョン管理 |

### 前提条件

- **AEM Forms as a Cloud Service**: コアコンポーネントがデフォルトで有効になっています
- **AEM 6.5 Forms**:AEM アーキタイプを使用して有効にする必要があります
- **ユーザー権限**: ユーザーは`forms-users` グループに属している必要があります
- **テンプレート**：アダプティブフォーム（コアコンポーネント）テンプレートが必要です
- **テーマ**: キャンバステーマ （OOTB）またはカスタムテーマ

### 制限事項

- **JSON スキーマの制約**:Null型、和集合タイプ （any）、OneOf/AnyOf/AllOf/NOT構文はサポートされていません
- **コンポーネントギャップ**:Adobe Sign ブロック、手書き署名、グラフ、画像の選択が利用できません（基盤コンポーネントで使用可能）
- **カスタム関数**：ジェネレーター関数、非同期/待機、クラスメソッドはサポートされていません
- **デジタル署名**：ネイティブでは利用できません（基盤コンポーネントとは異なります）

### 用途

**おすすめ対象：**

- 新しいフォーム開発プロジェクト
- 最新の保守が可能なアーキテクチャを必要とする企業
- 柔軟な公開が必要なプロジェクト（AEM + Edge Delivery + ヘッドレス）
- ヘッドレス APIによるカスタムフロントエンドアプリケーション（React、Angular）
- パフォーマンスとアクセシビリティを優先するプロジェクト
- オムニチャネル配信の要件（web、モバイル、キオスク）

**次の製品はお勧めできません：**

- Adobe Signとの統合が必要なForms（基盤コンポーネントを使用）
- Edge Delivery Servicesのパフォーマンスが重要なシンプルなフォーム
- 既存の基盤コンポーネントベースのフォーム（移行しない限り、基盤に維持）

## Edge Delivery Services Forms

### 定義

Edge Delivery Services（EDS） Formsは、Adobe Experience Manager Edge Delivery Servicesを通じてフォームを作成および配信するための構成可能なサービスのセットです。 エッジベースの配信、クライアントサイドのレンダリング、複数のオーサリングメソッドなどにより、優れたパフォーマンスを備えたフォームを迅速に開発できます。

### アーキテクチャ

EDS Formsは、次のような分離型エッジファーストのアーキテクチャを採用しています。

- **コンテンツソース**:Microsoft SharePoint、Google Drive （ドキュメントベース）またはユニバーサルエディター（WYSIWYG）
- **コードリポジトリ**: GitHub
- **コンテンツ配信**: Edge Delivery Services CDN
- **レンダリング**: Vanilla JavaScriptを使用したクライアント側レンダリング
- **フォームブロック**：アダプティブForms ブロックは、フォーム定義を処理し、HTMLを生成します

### レンダリングモデル

EDS Formsでは、クライアントサイドレンダリングのみを使用します。

1. JSONとして保存されたフォーム定義（スプレッドシートから変換するか、ユニバーサルエディターで作成）
2. エッジからJSONを取得しました：`https://<branch>--<repo>--<owner>.aem.live/<form>.json`
3. アダプティブ Forms ブロック JavaScriptはJSONを処理します
4. ブラウザーで動的に生成されるHTML構造
5. スタイル設定に適用されたCSS

**HTML構造パターン：**

```html
<div class="{Type}-wrapper field-{Name} field-wrapper" data-required="{Required}">
   <label for="{FieldId}" class="field-label">Label</label>
   <input type="{Type}" id="{FieldId}" name="{Name}">
   <div class="field-description" id="{FieldId}-description">Help text</div>
</div>
```

### オーサリング方法

EDS Formsでは、次の2つのオーサリングアプローチをサポートしています。

#### ユニバーサルエディター（WYSIWYG）

- ドラッグ&amp;ドロップ操作による視覚的なインターフェイス
- デバイスシミュレーションによるリアルタイムプレビュー
- 条件付きロジックの高度なルールエディター
- フォームデータモデル（FDM）の統合
- AEM Workflowsとの統合
- カスタムコンポーネントのサポート
- テンプレートベースの制作

#### ドキュメントベースのオーサリング

- Microsoft ExcelまたはGoogle Sheetsのオーサリング
- スプレッドシートベースのフォーム定義
- 即時の公開（変更内容をすぐに反映）
- シンプルから中程度の複雑なフォームに適しています

### 送信オプション

**Forms Submission Service （FSS）:**

- Google シートへの送信
- Microsoft Excel （OneDrive/SharePoint）に送信
- メール通知

**AEM公開送信アクション：**

- REST エンドポイント
- AEM メールサービス
- フォームデータモデル
- AEM ワークフロー
- SharePoint/OneDrive
- Azure Blob Storage
- Microsoft Power Automate
- Adobe Workfront Fusion
- Adobe Marketo Engage

### 主な機能

| カテゴリ | 機能 |
|----------|-------------|
| **コンポーネント** | すべてのHTML5入力タイプ、チェックボックスグループ、ラジオグループ、ドロップダウン、パネル、繰り返し可能なセクション、添付ファイル、アコーディオン、ウィザード、モーダル |
| **検証** | フィールドレベルの検証（必須、最小/最大、パターン）、カスタム検証メッセージ |
| **ルール** | 条件付き可視性、値式、イベント駆動型ルール |
| **統合** | Adobe Sign、Salesforce、Microsoft Dynamics、A/B テスト |
| **セキュリティ** | reCAPTCHA Enterprise、Captcha、CORS設定 |
| **Analytics** | Adobe Experience Platform Web SDK、フォームビューおよび送信トラッキング |

### 前提条件

**ドキュメントベースのオーサリングの場合：**

- GitHub アカウント
- Google ドライブまたはMicrosoft SharePoint アカウント
- ローカル開発用Node/npm
- アダプティブForms ブロックが設定されたAEM プロジェクト
- `forms@adobe.com`さんとフォルダーを共有（権限の編集）

**ユニバーサルエディター：**

- AEM Forms as a Cloud Service インスタンス
- Edge Delivery Services プロジェクトが設定されている
- ターゲット宛先に必要な権限

**AEM パブリッシュの送信用：**

- AEM Cloud Service インスタンス URLが設定されています
- OSGi リファラーフィルター設定
- EDGE DELIVERY サイトドメインのCORS設定

### 制限事項

**ドキュメントベースのオーサリング：**

- シートの命名制限：`helix-default`および`shared-aem`
- 複雑さが少ないフォームに最適
- 限定的なルール機能
- AEM ワークフローなし（AEM パブリッシュの送信なし）
- フォームデータモデルの統合なし

**ユニバーサルエディター：**

- カスタム関数で静的/動的インポートはサポートされていません
- フォームフラグメントはスタンドアロンでのみ編集できます
- 開発におけるフォーム埋め込み機能

**全般：**

- `shared-aem` シートにPII （一般に公開されているファイル）を含めることはできません
- クロスオリジン送信にCORS設定が必要です
- AEM パブリッシュの送信に必要なOSGi リファラーフィルター

### 用途

**おすすめ対象：**

- 高いLighthouse スコアを必要とする、パフォーマンスに不可欠なweb サイト
- すでにEdge Delivery Servicesを利用中のサイト
- 迅速なフォーム開発とデプロイメント
- 複雑さの単純から中程度のデータキャプチャ
- スプレッドシートベースのオーサリングを好むチーム（ドキュメントベース）
- 市場投入までの時間の短縮が必要なプロジェクト

**次の製品はお勧めできません：**

- 大規模なサーバーサイド処理を必要とするForms
- REST APIを欠くレガシーシステムとの緊密な統合
- オフラインフォームの要件
- JavaScriptの特定のフレームワーク（React、Angular）を必要とする企業は、コンテンツを詳細に制御できます

## 基盤コンポーネントに基づくアダプティブ Forms

### 定義

基盤コンポーネントは、従来のAEM Forms オーサリングアプローチを表しています。 これは、長年にわたってAEM Formsで使用されてきたオリジナルのアダプティブ Forms コンポーネントです。 Adobeでは、まだサポートされている間、新しいフォームを作成するのではなく、既存のフォームを維持するために主に基盤コンポーネントを推奨します。

### アーキテクチャ

基盤コンポーネントでは、従来のAEMアーキテクチャを使用しています。

- **コンテンツモデル**: JCR コンテンツ構造を持つWCM `cq:Page` コンポーネント
- **ルート構造**: `guideContainer` （ルート） → `rootPanel` → `items` （フォームフィールド）
- **レンダリング**: クライアントサイド JavaScriptを使用したサーバーサイドのレンダリング
- **SOM式**: フォームオブジェクトを参照するためのスクリプトオブジェクトモデル

### レンダリングモデル

基盤コンポーネントでは、サーバーサイドレンダリングを使用します。

1. JCR リポジトリに保存されたフォームコンテンツ
2. AEM server プロセスのフォーム構造
3. HTMLのレンダリングとブラウザーへの送信を完了
4. クライアントサイドのJavaScriptは、インタラクティブな機能と検証を処理し

### 主な機能

| カテゴリ | 機能 |
|----------|-------------|
| **コンポーネント** | すべてのコアコンポーネントに対応する40以上のコンポーネントに加えて、Adobe Sign ブロック、チャート、手書き署名、画像選択、サマリーステップ、ファイル添付リスト |
| **フォームモデル** | フォームデータモデル（FDM）、XDP/XFA テンプレート、XML スキーマ（XSD）、JSON スキーマ |
| **ルールエンジン** | ビジュアルエディター+ コードエディター（forms-power-users グループ用） |
| **デジタル署名** | Adobe Signとの統合、手書き署名 |
| **送信アクション** | コアコンポーネントと類似した複数のOOTB アクション |

### 基盤専用のコンポーネント

次のコンポーネントは、**基盤コンポーネントでのみ使用できます**:

- Adobe Sign ブロック
- グラフ
- ファイル添付リスト
- 脚注プレースホルダー
- 画像選択
- 手書き署名
- 概要ステップ
- Turnstile Captcha

### 前提条件

- AEM Forms as a Cloud ServiceまたはAEM 6.5 Forms
- アダプティブフォームテンプレート（基礎）
- `forms-users` グループのユーザー

### 制限事項

- **公開**: AEMのみ（Edge Delivery Servicesまたはヘッドレス APIなし）
- **パフォーマンス**：標準パフォーマンス（Lighthouse スコア用に最適化されていません）
- **アーキテクチャ**: BEMに準拠しないレガシーアーキテクチャ
- **Sourceを開く**: オープンソースではありません
- **将来開発**: メンテナンスモード。主にコアコンポーネントに追加される新機能

### 用途

**おすすめ対象：**

- 既存の基盤ベースのフォームを維持する
- Adobe Signまたは手書き署名の統合を必要とするForms
- 確立された基盤の機能に依存するワークフロー
- AEMのみの公開要件を持つプロジェクト
- Foundation専用のコンポーネント（チャート、画像の選択）を必要とするForms

**次の製品はお勧めできません：**

- 新しいフォーム開発（コアコンポーネントを使用）
- Edge Delivery Servicesの公開を必要とするプロジェクト
- ヘッドレスフォーム API
- パフォーマンスに不可欠な実装
- 将来性のあるアーキテクチャを優先するプロジェクト

| ヘッドレス/オムニチャネル配信 | コアコンポーネント |

| スプレッドシートベースのオーサリング | Edge Delivery Services （ドキュメントベース） |
| Edge DeliveryによるVisual WYSIWYGのオーサリング | Edge Delivery Services（ユニバーサルエディター） |
| 複雑な統合機能 | コアコンポーネントまたは基盤コンポーネント |
| 迅速なプロトタイピング | Edge Delivery Services （ドキュメントベース） |
| XFA/XDP テンプレートのサポート | 基盤コンポーネント |
| カスタム React/Angular フロントエンド | コアコンポーネント（ヘッドレス API） |

## 移行の考慮事項

### 基盤コンポーネントからコアコンポーネントへ

- **ツール**: AEM Modernize Tools （Forms Conversion Utility）
- **労力**: フォームの複雑さに応じて中～高程度
- **考慮事項**:
   - ルールを手動で作成し直す必要がある
   - Foundation専用コンポーネント（Adobe Sign ブロック、手書き署名、グラフ）は、移行時に削除されます
   - 翻訳設定が引き継がれません
   - カスタム関数には書き換えが必要です

### Edge Delivery Servicesの従来型CMS

- **アプローチ**：ユニバーサルエディターまたはドキュメントベースのオーサリングを使用したフォームの再構築
- **労力**：複雑なフォームでは高、単純なフォームでは低
- **メリット**：大幅なパフォーマンス向上、最新の開発エクスペリエンス

## ドキュメントのギャップと曖昧さ

次の領域のドキュメントが不完全または曖昧です。

1. **コアコンポーネント XDP/XFA サポート**: ドキュメントでは「制限付き」サポートが示されていますが、厳密な制限は指定されていません
2. **Edge Delivery Services フォーム埋め込み**：ドキュメントで「近日公開予定」とマークされています。現在のステータスは不明です
3. **基盤コンポーネントの将来**：明示的な非推奨タイムラインがありません。「メンテナンスモード」と説明されます。
4. **ヘッドレス API カバレッジ**: サーバーレンダリングされたフォームとヘッドレスのフォームの間の正確な機能パリティは文書化されていません
5. **パフォーマンスベンチマーク**：特定のLighthouse スコアのアプローチ間の比較が提供されていません
