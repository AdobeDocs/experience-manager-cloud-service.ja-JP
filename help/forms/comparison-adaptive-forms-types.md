# アダプティブForms：コアコンポーネントとEdge Delivery Services Formsの基盤コンポーネントの比較

Adobe Experience Manager（AEM）Formsは、データキャプチャエクスペリエンスを構築するために、コアコンポーネントに基づくアダプティブForms、Edge Delivery Services Forms、基盤コンポーネントに基づくアダプティブFormsの 3 つの異なるアプローチを提供します。 アプローチごとにアーキテクチャ、レンダリングモデル、ターゲットのユースケースが異なります。 この記事では、ソリューションアーキテクト、開発者、AEMのお客様が要件に合わせて適切なアプローチを選択するのに役立つ、技術的な比較を説明します。

## 概要

3 種類のフォームタイプはすべて、ユーザーデータを取得し、バックエンドシステムと統合する目的に使用します。 ただし、基盤となるアーキテクチャ、フォームのレンダリング場所、フォームの配信方法、サポートする機能はそれぞれ異なります。

| アプローチ | ステータス | プライマリのユースケース |
|----------|--------|------------------|
| **コアコンポーネント** | 新しいフォームにお勧め | 柔軟な公開オプションを備えたAEM オーサリングが必要な、最新のスケーラブルなフォーム |
| **Edge Delivery ServicesForms** | パフォーマンスが重要なサイトに推奨 | 迅速なデプロイメントでエッジから配信される高パフォーマンスのフォーム |
| **基盤コンポーネント** | メンテナンスモード | 従来の機能のサポートを必要とする既存のフォーム |

## コアコンポーネントに基づくアダプティブForms

### 定義

アダプティブ Forms コアコンポーネントは、Adobe Experience Manager WCM コアコンポーネントの基盤上に構築された、BEM 準拠の 30 個のオープンソース コンポーネントです。 これらは、新しいアダプティブFormsを作成し、最新のアーキテクチャ、パフォーマンスの向上、自動ヘッドレスフォームの生成を実現するためにAdobeが推奨するアプローチを表しています。

### アーキテクチャ

コアコンポーネントは、標準化されたモジュール型コンポーネントアーキテクチャを使用します。

- **コンポーネントの基盤**:AEM WCM コアコンポーネントに基づいて構築
- **スタイル設定方法**:BEM （ブロック要素修飾子） CSS 規則
- **コンテンツ保存場所**：構造化コンテンツノードを含む JCR リポジトリ
- **レンダリング**：オプションのクライアントサイドヘッドレスレンダリングを使用した、AEMでのサーバーサイドレンダリング
- **Source**: オープンソース（.GitHub[ で入手可能 ](https://github.com/adobe/aem-core-forms-components)

### レンダリングモデル

コアコンポーネントは、次のような複数のレンダリングモデルをサポートします。

1. **サーバーサイドレンダリング（SSR）**:AEM サーバーでFormsをレンダリングし、HTML全体をブラウザーに配信します
2. **ヘッドレスレンダリング**:Formsは、React、Angular、その他のフレームワークでのクライアントサイドレンダリング用の API を使用して JSON 表現を公開します
3. **ハイブリッドレンダリング**：最適なパフォーマンスを実現するための、サーバーとクライアントのレンダリングの組み合わせ

### オプションを公開中

- AEM パブリッシュインスタンス
- Edge Delivery Services（設定の場合）
- カスタムフロントエンドアプリケーション用のヘッドレス API

### 主な機能

| カテゴリ | 機能 |
|----------|-------------|
| **コンポーネント** | テキストボックス、数値ボックス、日付選択、ドロップダウン、チェックボックスグループ、ラジオボタン、ファイル添付、ウィザード、アコーディオン、水平/垂直タブなど 30 個の標準化されたコンポーネント |
| **フォームモデル** | JSON スキーマ（v4 および 2020-12）、フォームデータモデル（FDM）、XDP/XFA テンプレート（制限付き） |
| **ルールエンジン** | 条件付きロジック、検証、サービス呼び出し、カスタム関数を含むビジュアルルールエディター |
| **送信アクション** | REST エンドポイント，メール，AEM ワークフロー，SharePoint, OneDrive, Azure Blob Storage, Power Automate, Workfront Fusion, フォームデータモデル |
| **事前入力** | フォームデータモデル事前入力サービス、カスタム事前入力サービス |
| **CAPTCHA** | reCAPTCHA, hCaptcha, Turnstile |
| **レコードのドキュメント** | カスタムテンプレートまたは OOTB テンプレートを使用したPDFの生成 |
| **アクセシビリティ** | WCAG 準拠、ARIA ラベル、キーボードナビゲーション、スクリーンリーダーのサポート |
| **ローカライゼーション** | AEM翻訳ワークフローによる多言語サポート |
| **バージョン管理** | AEM Sitesから継承されたコンテンツのバージョン管理 |

### 前提条件

- **AEM Forms as a Cloud Service**: コアコンポーネントはデフォルトで有効になっています
- **AEM 6.5 Forms**: AEM アーキタイプを使用して有効にする必要があります
- **ユーザー権限**: ユーザーは `forms-users` グループに属している必要があります
- **テンプレート**：アダプティブフォーム（コアコンポーネント）のテンプレートは必須です
- **テーマ**：キャンバステーマ（OOTB）またはカスタムテーマ

### 制限事項

- **JSON スキーマ制約**:Null 型、和集合型（any）、OneOf/AnyOf/AllOf/NOT 構成はサポートされていません
- **コンポーネントのギャップ**:Adobe Sign ブロック、手書き署名、グラフ、画像の選択はできません（基盤コンポーネントで使用可能）
- **カスタム関数**: ジェネレーター関数、async/await、クラスメソッドがサポートされていません
- **電子署名**：ネイティブでは使用できません（基盤コンポーネントとは異なります）

### 用途

**推奨**

- 新規フォーム開発プロジェクト
- 最新のメンテナンス可能なアーキテクチャを必要とする組織
- 柔軟な公開が必要なプロジェクト（AEM + Edge Delivery + ヘッドレス）
- ヘッドレス API を介したカスタムフロントエンドアプリケーション（React、Angular）
- パフォーマンスとアクセシビリティを優先するプロジェクト
- オムニチャネル配信要件（web、モバイル、キオスク）

**次の場合は推奨されません：**

- Adobe Sign 統合が必要なForms（基盤コンポーネントを使用）
- Edge Delivery Servicesのパフォーマンスが重要なシンプルなフォーム
- 既存の基盤コンポーネントベースのフォーム（移行しない限り、基盤の中で維持します）

## Edge Delivery Services Forms

### 定義

Edge Delivery Services（EDS）Formsは、Adobe Experience Manager Edge Delivery Servicesを通じてフォームを作成して配信するための、構成可能なサービスセットです。 エッジベースの配信、クライアントサイドレンダリング、複数のオーサリング方法を通じて、卓越したパフォーマンスで迅速なフォーム開発を可能にします。

### アーキテクチャ

EDS Formsは、切り離されたエッジ・ファースト・アーキテクチャを使用します。

- **コンテンツソース**:Microsoft SharePoint、Google Drive （ドキュメントベース）または Universal Editor （WYSIWYG）
- **コードリポジトリー**:GitHub
- **コンテンツ配信**:Edge Delivery Services CDN
- **レンダリング**:vanilla JavaScriptを使用したクライアントサイドレンダリング
- **フォームブロック**: アダプティブ Forms ブロックは、フォーム定義を処理し、HTMLを生成します

### レンダリングモデル

EDS Formsは、クライアントサイドレンダリングのみを使用します。

1. JSON 形式で保存されたフォーム定義（スプレッドシートから変換するか、ユニバーサルエディターで作成）
2. エッジから取得された JSON: `https://<branch>--<repo>--<owner>.aem.live/<form>.json`
3. Adaptive Forms ブロック JavaScriptは JSON を処理します
4. ブラウザーで動的に生成されるHTML構造
5. スタイル設定に適用された CSS

**HTML構造パターン：**

```html
<div class="{Type}-wrapper field-{Name} field-wrapper" data-required="{Required}">
   <label for="{FieldId}" class="field-label">Label</label>
   <input type="{Type}" id="{FieldId}" name="{Name}">
   <div class="field-description" id="{FieldId}-description">Help text</div>
</div>
```

### オーサリング方法

EDS Formsは、次の 2 つのオーサリング・アプローチをサポートします。

#### ユニバーサルエディター（WYSIWYG）

- 視覚的なドラッグ&amp;ドロップインターフェイス
- デバイスシミュレーションを使用したリアルタイムプレビュー
- 条件付きロジック用の高度なルールエディター
- フォームデータモデル（FDM）統合
- AEM ワークフローの統合
- カスタムコンポーネントのサポート
- テンプレートベースの作成

#### ドキュメントベースのオーサリング

- Microsoft Excel またはGoogle シートのオーサリング
- スプレッドシートベースのフォーム定義
- 即時公開（変更は直ちに反映されます）
- 単純なフォームから中程度の複雑なフォームに適しています

### 送信オプション

**Forms送信サービス （FSS）:**

- Google Sheets への送信
- Microsoft Excel への送信（OneDrive/SharePoint）
- メール通知

**AEM公開送信アクション：**

- REST エンドポイント
- AEM メールサービス
- フォームデータモデル
- AEM ワークフロー
- SharePoint/OneDrive
- Azure Blob Storage
- Microsoft Power Automate
- Adobe Workfront Fusion
- Adobe Marketo Engage

### 主な機能

| カテゴリ | 機能 |
|----------|-------------|
| **コンポーネント** | すべてのHTML5 入力タイプ、チェックボックスグループ、ラジオグループ、ドロップダウン、パネル、繰り返し可能なセクション、ファイル添付、アコーディオン、ウィザード、モーダル |
| **検証** | フィールドレベルの検証（必須、最小/最大、パターン）、カスタム検証メッセージ |
| **ルール** | 条件表示、値式、イベント駆動型ルール |
| **統合** | Adobe Sign、Salesforce、Microsoft Dynamics、A/B テスト |
| **セキュリティ** | reCAPTCHA Enterprise、hCaptcha、CORS 設定 |
| **Analytics** | Adobe Experience Platform Web SDK、フォームビュー、送信トラッキング |

### 前提条件

**ドキュメントベースのオーサリングの場合：**

- GitHub アカウント
- Google Drive またはMicrosoft SharePoint アカウント
- ローカル開発用の Node/npm
- アダプティブAEMブロックが設定されたForms プロジェクト
- `forms@adobe.com` とフォルダーを共有（編集権限）

**ユニバーサルエディターの場合：**

- AEM Forms as a Cloud Service インスタンス
- Edge Delivery Services プロジェクトが設定されている
- ターゲット宛先に必要な権限

**AEMへの公開送信用：**

- AEM Cloud Service インスタンスの URL が設定されました
- OSGi リファラーフィルターの設定
- Edge Delivery サイトドメインの CORS 設定

### 制限事項

**ドキュメントベースのオーサリング：**

- シート名は `helix-default` と `shared-aem` に制限されています
- 複雑さの低いフォームに最適
- 制限付きルールの機能
- AEM ワークフローなし（AEM公開送信を除く）
- フォームデータモデルの統合がありません

**ユニバーサルエディター：**

- 静的/動的インポートは、カスタム関数ではサポートされていません。
- フォームフラグメントはスタンドアロンでのみ編集できます
- 開発時のフォーム埋め込み機能

**一般：**

- `shared-aem` シートには PII （公開アクセス可能）を含めない
- クロスオリジン送信に CORS 設定が必要
- AEMのパブリッシュ送信に必要な OSGi リファラーフィルター

### 用途

**推奨**

- 高い Lighthouse スコアを必要とするパフォーマンスクリティカルな web サイト
- 既にEdge Delivery Servicesを使用しているサイト
- フォームの迅速な開発とデプロイ
- シンプルから中程度の複雑さのデータキャプチャ
- スプレッドシートベースのオーサリングを好むチーム（ドキュメントベース）
- 迅速な市場投入が必要なプロジェクト

**次の場合は推奨されません：**

- 大規模なサーバーサイド処理を必要とするForms
- REST API が不足しているレガシーシステムとの深い統合
- オフラインフォームの要件
- フルコントロールを使用した特定のJavaScript フレームワーク（React、Angular）を必要とする組織

## 基盤コンポーネントに基づくアダプティブForms

### 定義

基盤コンポーネントは、従来のAEM Forms オーサリングアプローチを表します。 これらは、AEM Formsで何年も使用されてきた、元のアダプティブ Forms コンポーネントです。 Adobeでは、引き続きサポートされますが、新しいフォームを作成するのではなく、主に既存のフォームを維持するための基盤コンポーネントをお勧めします。

### アーキテクチャ

基盤コンポーネントでは、従来のAEM アーキテクチャを使用します。

- **コンテンツモデル**:JCR コンテンツ構造を持つ WCM `cq:Page` コンポーネント
- **ルート構造**:`guideContainer` （ルート）→`rootPanel` → `items` （フォームフィールド）
- **レンダリング**：クライアントサイド JavaScriptを使用したサーバーサイドレンダリング
- **SOM 式**: フォームオブジェクトを参照するためのスクリプティングオブジェクトモデル

### レンダリングモデル

基盤コンポーネントでは、サーバーサイドレンダリングを使用します。

1. JCR リポジトリに保存されたフォームコンテンツ
2. AEM サーバーがフォーム構造を処理する
3. レンダリングされブラウザーに送信されるHTMLの完成
4. クライアントサイド JavaScriptでインタラクティビティと検証を処理

### 主な機能

| カテゴリ | 機能 |
|----------|-------------|
| **コンポーネント** | すべてのコアコンポーネントと同等の機能を含む 40 を超えるコンポーネントと：Adobe Sign ブロック、グラフ、手書き署名、画像選択、概要ステップ、ファイル添付ファイルのリスト |
| **フォームモデル** | フォームデータモデル（FDM）、XDP/XFA テンプレート、XML スキーマ（XSD）、JSON スキーマ |
| **ルールエンジン** | ビジュアルエディター+ コードエディター（forms-power-users グループの場合） |
| **デジタル署名** | Adobe Sign 統合、手書き署名 |
| **送信アクション** | コアコンポーネントと類似した複数の OOTB アクション |

### Foundation 専用のコンポーネント

次のコンポーネントは、基盤コンポーネントで **のみ使用可能** です。

- Adobe Sign ブロック
- グラフ
- ファイル添付リスト
- 脚注プレースホルダー
- 画像選択
- 手書き署名
- 概要ステップ
- Turnstile Captcha

### 前提条件

- AEM Forms as a Cloud ServiceまたはAEM 6.5 Forms
- アダプティブフォームテンプレート （基盤）
- グループ内 `forms-users` ユーザー

### 制限事項

- **公開**:AEMのみ（Edge Delivery Services API またはヘッドレス API を除く）
- **パフォーマンス**：標準パフォーマンス（Lighthouse スコア用に最適化されていません）
- **アーキテクチャ**:BEM コンプライアンスを使用しないレガシーアーキテクチャ
- **Sourceを開く**：オープンソースではない
- **今後の開発**：メンテナンスモード、主にコアコンポーネントに追加される新機能

### 用途

**推奨**

- 既存の基盤ベースのフォームの維持
- Adobe Sign または手書き署名統合が必要なForms
- 確立された基盤機能に応じたワークフロー
- AEMのみの公開要件を持つプロジェクト
- Formsには基盤専用のコンポーネント（グラフ、画像選択）が必要

**次の場合は推奨されません：**

- 新しいフォーム開発（コアコンポーネントを使用）
- Edge Delivery Servicesでの公開が必要なプロジェクト
- ヘッドレスフォーム API
- パフォーマンス・クリティカルな実装
- 将来性のあるアーキテクチャを優先するプロジェクト

## 決定マトリックス

| 要件 | 推奨されるアプローチ |
|-------------|---------------------|
| 新しいフォームの開発 | コアコンポーネント |
| 既存フォームの保守 | 基盤コンポーネント（移行まで） |
| パフォーマンスクリティカル（Lighthouse のスコアが高い） | Edge Delivery Services Forms |
| ヘッドレス/オムニチャネル配信 | コアコンポーネント |
| Adobe Sign の統合 | 基盤コンポーネント |
| スプレッドシートベースのオーサリング | Edge Delivery Services（ドキュメントベース） |
| Edge 配信を使用したビジュアル WYSIWYG オーサリング | Edge Delivery Services（ユニバーサルエディター） |
| 複雑なエンタープライズ統合 | コアコンポーネントまたは基盤コンポーネント |
| ラピッドプロトタイピング | Edge Delivery Services（ドキュメントベース） |
| XFA/XDP テンプレートのサポート | 基盤コンポーネント |
| カスタム React/Angular フロントエンド | コアコンポーネント（ヘッドレス API） |

## 移行に関する考慮事項

### 基盤コンポーネントからコアコンポーネントへ

- **ツール**:AEM Modernize Tools （Forms変換ユーティリティ）
- **作業量**：フォームの複雑さに応じて中程度から高い
- **考慮事項**:
   - ルールを手動で再作成する必要がある
   - 基盤専用コンポーネント（Adobe Sign ブロック、手書き署名、グラフ）は、移行中に削除されます
   - 引き継がれない翻訳設定
   - カスタム関数を書き換える必要がある

### 従来の形式からEdge Delivery Servicesへ

- **アプローチ**：ユニバーサルエディターまたはドキュメントベースのオーサリングを使用してフォームを再構築します
- **作業量**：複雑なフォームでは高く、単純なフォームでは低い
- **メリット**：大幅なパフォーマンス向上、最新の開発経験

## ドキュメントのギャップと曖昧さ

次の領域に、不完全またはあいまいなドキュメントがあります。

1. **コアコンポーネント XDP/XFA のサポート**：ドキュメントは「限定的」なサポートを示していますが、正確な制限を示していません
2. **Edge Delivery Services フォームの埋め込み**: ドキュメントで「近日公開予定」とマークされています。現在のステータスは不明です
3. **基盤コンポーネントの将来**：明示的な廃止タイムラインがありません。「メンテナンスモード」と説明されます
4. **ヘッドレス API の対象範囲**：サーバーレンダリング フォームとヘッドレスフォームの間の正確な機能パリティはドキュメントではありません
5. **パフォーマンスベンチマーク**：アプローチ間の特定の Lighthouse スコア比較が提供されていません
